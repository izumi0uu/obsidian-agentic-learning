## 题目正文

### 1. 子问题：记忆系统与状态设计（短期/长期/State）

主问题：如何设计 [[Agent]] 的短期记忆、[[Long-term Memory|长期记忆]]和[[Agent State|运行状态]]？

口述答案：
短期记忆承载当前任务上下文、中间结果和工具返回，保证这次任务连贯；长期记忆承载跨会话偏好、稳定事实和历史经验。实现上一般是“短期窗口/摘要 + 长期[[Vector Database|向量库]]/结构化库”组合。State 设计要遵循最小化原则：只放决策必需字段，长文本放外部存储并在状态里保留引用 ID。再把状态拆成业务数据、过程数据、观测数据三层，防止节点互相污染与 [[Token|token]] 爆炸。

常见追问：

1. 如何做记忆写入策略，避免“写入噪声”？
  1. **分级写入，不直接入长期库**
    候选区，只有“高频出现 + 高置信 + 可复用”的信息才转正。
  2. **写前去重合并**
    语义相似度 + 关键字段”去重，已有就更新，不重复新增，防记忆膨胀和冲突。
  3. **记忆有生命周期**
    来源/时间/版本/TTL，长期不命中自动降权或清理，避免旧噪声长期污染。
2. 状态变大后如何控 token 成本？
  1. 第一是最小状态，状态只放决策必需信息，长文本走外部存储引用；第二是分层压缩，把历史和中间过程变成结构化摘要，避免上下文无限增长；第三是加预算硬阈值，超过 token 或步数就自动降级，比如缩小检索 topK 或关闭次要节点。这样能把成本和时延稳定在可控范围内。
3. 长期记忆如何做过期与纠错？
  1. **生命周期管理（过期）**
    长期记忆都带 TTL + 最后命中时间 + 重要度，长期不命中自动降权，超期归档/清理。
  2. **冲突检测与版本化（纠错）**
    事实写入时先做冲突检测（如“用户偏好A/B冲突”），不直接覆盖，走版本化：保留旧版、标记新版、可回滚。
  3. **验证闭环（纠错）**
    险记忆必须二次验证（多次出现、可信来源、用户确认），并提供“用户纠正入口”，纠正后优先覆盖旧结论。
