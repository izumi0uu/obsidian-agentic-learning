## 题目正文

### 高并发场景下的数据库连接池应该如何优化

1. **maxWait**
- 表示从连接池获取连接时最多等待多久。
- 如果默认无限等待，高并发下连接池一旦耗尽，大量请求会长期卡死在线程里，最后把应用线程池也拖死，服务呈现“假死”。
- 实践上通常会设置一个有限等待时间，而不是无限等。

2. **connectionTimeout 和 socketTimeout**
- `connectionTimeout`：建立 TCP 连接的超时时间；
- `socketTimeout`：发出请求后等待数据库响应的超时时间。
- 高并发场景下如果网络异常，这两个超时不配好，连接可能长期僵死，占着池子不释放，最终拖垮整个服务。

3. **maxActive / 最大连接数**
- 最大连接数不是越大越好。
- 如果连接数开得过大，数据库 CPU、上下文切换和锁竞争反而会更严重，整体吞吐可能下降。
- 工程上通常按数据库承载能力、SQL RT 和并发模型来压测后设定，不会无脑拉满。

面试里可以一句话总结：
**连接池调优不是“多给连接”，而是“限制等待、设置超时、控制上限”，避免应用线程和数据库一起被拖死。**
