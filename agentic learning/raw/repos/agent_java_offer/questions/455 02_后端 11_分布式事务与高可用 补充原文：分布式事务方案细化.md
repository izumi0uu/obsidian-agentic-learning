### 原文内容

### 分布式事务常见方案

1. **XA / 两阶段提交（2PC）**
- 事务管理器先询问每个参与方能否执行；
- 第一阶段预留资源，第二阶段统一提交或回滚；
- 一致性强，但严重依赖数据库层面的锁和协调，不适合高并发场景。

2. **TCC（Try / Confirm / Cancel）**
- Try：检测资源并预留；
- Confirm：真正执行业务；
- Cancel：失败时执行补偿回滚。
- 优点是可实现较强一致；缺点是业务侵入大、补偿逻辑复杂。
- 更适合资金类、对一致性要求很高、且每个环节执行时间较短的场景。

3. **本地消息表**
- A 系统在本地事务里同时写业务数据和消息表；
- 后台线程轮询消息表，再调用 B 系统；
- B 系统必须幂等。
- 缺点是高并发下消息表本身会成为数据库压力点。

4. **[[Durable Execution|可靠消息最终一致性]]**
- A 先向 MQ 发送 prepared / half message；
- 本地事务成功后再发 confirm / commit；
- 如果 confirm 丢了，MQ 通过回查机制再次确认本地事务状态；
- B 消费消息时需要幂等、失败重试和补偿。
- 这种方案更适合订单、库存、履约这类允许最终一致的链路。

### 订单场景里的可靠消息最终一致性

典型流程：
1. 先发 half message；
2. MQ 落盘成功后再走核心订单链路；
3. 核心链路成功则发 commit，失败则发 rollback；
4. MQ 在等待期间可通过回查机制确认事务状态；
5. 消费者收到消息后执行下游逻辑，失败则重试或进入补偿。

### Seata / TCC 框架理解

Seata 的角色通常分为：
- `TC`：协调器，维护全局事务状态；
- `TM`：事务发起者，负责开始/提交/回滚全局事务；
- `RM`：资源管理器，管理各分支事务并向 TC 汇报。

它的核心流程是：
- TM 向 TC 申请一个全局事务 ID；
- 各分支事务把自己注册到这个全局事务里；
- 成功则统一提交，失败则由 TC 驱动各分支回滚。

面试里可以补一句：
- 强一致核心链路更适合 TCC；
- 大部分互联网业务更常用“可靠消息 + 幂等 + 补偿 + 对账”这套最终一致方案。
