## 题目正文

### 你今晚最该准备的 8 个题眼

**1）Agent 架构怎么设计**
一定要能讲清：Planner / Executor / Tool Router / Memory / Observation / Evaluator / Retry / Fallback 这几层怎么分工；什么时候单 Agent，什么时候 Multi-Agent；多 Agent 的收益不是“更酷”，而是 **上下文隔离、专业化和并发**。这点和 某AI应用平台 官方对 Sub-agent 的说法完全一致。

**2）RAG / 知识库 / memory**
至少要能顺下来：文档清洗 → chunk 策略 → 索引 → recall → rerank → prompt assembly → citation / grounding。然后能回答：为什么要 multi-query、是否要二次筛选、embedding 检索和规则知识库有什么差异、错知识/过期知识怎么治理。公开面经里这些都被直接问到了。

**3）大代码库里如何让 AI 理解项目**
这题很可能会打到你。你要准备一套像样答案：目录图谱、模块摘要、rules/约束、关键[[Trace|调用链]]、核心 domain object、增量索引、变更影响分析、只给局部上下文而不是整仓塞满。公开面经已经出现了“几十万行代码项目里怎么让 AI 快速理解结构”。

**4）AI 出码为什么会错，怎么提高留用率**
你要能讲“看起来对但逻辑错”的根因：上下文缺失、需求不清、约束没喂、检索命中不准、工具执行反馈不够、评测闭环缺失。然后讲治理动作：任务切小、先影响分析、规则库、检索、计划先行、双 Agent review、单测/集测、灰度/双跑。面经里直接问到了出码留用率、复杂业务质量下降原因、错知识误导。

**5）MCP / tools / workflow orchestration**
某AI应用平台 官方文档和岗位要求都提到了 MCP 和大量工具集成。你至少要能回答：MCP 解决什么问题、和传统 tool calling 的关系、权限/鉴权怎么做、工具结果如何标准化、失败重试和超时如何处理。

**6）评测与质量[[Observability|监控]]**
不是只会说 accuracy。你要讲离线和在线两套：任务成功率、步骤成功率、工具调用成功率、引用正确率、用户采纳率、人工复核率、延迟、token 成本、回退率、重试率。某AI应用平台 公开面经里有“如何建立 AI 应用质量监控体系”，官方也明显把“靠谱”和“交付结果”放在核心位置。

**7）灰度、双跑、线上稳定性**
公开面经里已经问到“灰度切流 + 在线双跑”。这是二面很喜欢的工程味问题。你要能讲：新旧链路如何并行、结果怎么比对、日志和埋点怎么做、怎么降级回滚、如何避免影响真实用户。

**8）基础功不能空**
Transformer、HNSW、Redis、缓存一致性、Linux/Git、网络基础、一道中等算法题，这些不能完全不备。某AI应用平台 的 AI 岗公开面经里这些东西反复出现。

你可以优先把下面 12 题过一遍：

1. 你做的 Agent 系统里，**规划、执行、工具调用、memory** 分别怎么设计？
2. **Single Agent 和 Multi-Agent** 你怎么选？什么场景必须拆？
3. **Sub-agent** 的收益到底是什么，除了“并行”还有什么？
4. **上下文隔离**

每个子代理只看自己需要的信息，减少主上下文污染、幻觉和 prompt 变长问题。

1. **职责专业化**

可以按角色拆分（检索、执行、审校、风控），每个子代理用不同提示词/工具/模型，更稳。

1. **故障隔离**

某个子代理跑偏或失败，不会拖垮整条链路；可以局部重试、局部回滚。

1. **权限最小化**

给不同子代理不同权限（只读/可写/高危工具禁用），安全边界更清晰。

1. **成本与性能分层**

简单任务给小模型，关键步骤给强模型，实现“该省省、该强强”。

1. 知识库/RAG 里，**chunk、召回、rerank、multi-query** 你怎么做？
2. **规则知识库** 和 **embedding memory** 的差别是什么？

`规则知识库` 和 `embedding memory` 的差别可以记成：**硬规则 vs 软记忆**。

- 规则库：稳定、可审计、可追责（适合合规/风控）。  
- embedding：覆盖广、能处理长尾，但可能召回偏差。

 **适用场景**  

- 规则库：风控边界、权限、计费、交易限制。  
- embedding：FAQ、经验复用、个性化上下文、知识问答。

**工程实践**  

  **规则库做“硬约束和最终裁决”，embedding 做“语义检索和补充上下文”。**

1. 如果知识库内容**过期或错误**，怎么避免误导模型？

先说入库，所有知识都必须带版本号、生效/失效时间、来源等级和责任人，过期内容自动降权或下线；像费率、风控、政策这类高风险知识要人工审核后才能进入高可信索引。

第二层是检索，先做 metadata 硬过滤，只查有效时间内、可信来源的数据，再做向量+关键词混检和重排序，把相似度、新鲜度、可信度一起打分，避免旧文档因语义相近被误召回。

第三层是生成，要求模型“有证据才回答”，必须带引用；证据不足或冲突时直接拒答或提示[[Human-in-the-loop|人工确认]]。

最后是线上止损，快速封禁问题文档ID或来源, 持续监控过期命中率、引用覆盖率、冲突率、幻觉率，指标异常就一键切高可信索引并回滚快照。

1. 在一个**几十万行代码仓库**里，你怎么让 AI 快速理解项目结构？

我不会让 AI 一上来就读几十万行代码，这样又慢又乱。

第一步先把项目画成目录/地图：哪是入口、哪是核心业务、哪是数据库、哪是外部接口。先让 AI 知道“这个系统大概怎么跑”。

第二步再做“按需找文件”，比如要改登录，就只给它登录相关的 5 到 10 个文件，不会把全仓库都塞进去。

第三步是先看主流程，比如一个请求怎么从 Controller 走到 Service，再到数据库，这样 AI 很快能抓住主干，不会在细节里迷路。

第四步是边看边验证，AI 说某个类负责某功能，我会让它用搜索和调用关系再确认一下. 最后把确认过的结论沉淀成小文档，比如“模块说明、关键入口、常见坑”，下次再问就更快。

核心思路就一句话：**先搭框架，再看细节；渐进式披露相关内容；每一步都要验证以及沉淀摘要。** 

1. 为什么你们不是让 AI **直接从 PRD 出码**，而是“方案驱动”？
2. AI 出码“**看起来对但逻辑错**”时，你怎么发现、怎么兜底？

第一步写代码时引入spec规范约束, 主要是给 AI 做一些硬性的不能做的约束。引入一些正确的范例，以及团队的代码规约。 测试时需求拆成可验证的用例，至少覆盖三类：正常场景、边界场景和异常场景，比如空参数、重复请求、并发冲突、超时重试这些最容易出问题的点。

第二步我会做小流量灰度，不只看接口 200，还看真实业务指标，比如成功率、错误率、延迟、资金和库存是否异常，确认逻辑在真实流量下也成立。

第三步是兜底机制，新逻辑必须挂 Feature Flag，发现异常可以一键关闭并切回旧逻辑；写操作加幂等键，避免重复扣费或重复下单；关键链路保留[[Audit Log|审计日志]]和 request_id，出问题可以快速回放定位。

1. **MCP / tool calling** 如何做权限控制、失败重试和结果标准化？

我们做 MCP / tool calling，不让模型直接“自由调用”，而是经过一个 Tool Gateway。

第一层是权限控制：按用户、Agent、场景做 RBAC/ABAC，默认[[Least Privilege Tools|最小权限]]，只开放白名单工具；高危工具（下单、转账、删数据）必须二次确认，参数也做 schema 校验和额度限制。

第二层是失败重试：先做错误分类，只有超时、限流、临时 5xx 这类可恢复错误才重试；业务错误、参数错误、权限错误不重试。重试采用指数退避+抖动，设最大重试次数和总超时，配熔断与降级，防止雪崩。写操作必须幂等键，避免重试造成重复副作用。

第三层是结果标准化：所有工具返回统一 envelope，例如 `code/message/data/retryable/trace_id`，并把外部错误码映射成平台标准错误码；模型只消费标准字段，不直接依赖第三方原始响应。这样做的价值是：权限可控、重试可预期、结果可编排，也方便监控和审计。  

**模型只负责决策，工具网关负责安全与可靠性。**

1. Agent 应用怎么做**评测和线上监控**？

面试就按“**离线评测 + 线上监控 + 回归闭环**”讲。

**可背版（300-500字）**  

离线侧先做分层评测，第一层看任务成功率（是否完成目标），第二层看过程质量（工具调用是否正确、是否越权、是否死循环），第三层看安全与鲁棒性（注入攻击、脏数据、长尾问题）。数据集要覆盖真实用户样本、边界样本和故障样本，并保留一套黄金集做版本对比。

线上侧我会盯四类指标：效果指标（成功率、人工接管率、用户满意度）、效率指标（P50/P95延迟、步骤数、token成本）、稳定性指标（工具失败率、重试率、超时率、回环率）、安全指标（越权调用、拒答命中、敏感操作拦截）。同时全链路打 trace_id，把一次任务里“计划-调用-结果-最终答复”串起来，便于定位问题。治理上要有灰度发布、实时告警和一键降级，异常时先止损再排查。

最后把线上 bad case 回流到离线评测集，持续做 A/B 和回归测试。
