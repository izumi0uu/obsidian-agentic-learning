## 题目正文

### 2. [[Durable Execution|最终一致性]]常见模式？

1. Outbox 事件模式  
本地业务数据和“待发送事件”写在同一个数据库事务里。  
比如订单服务创建订单时，同时写一条 order_created 到 outbox 表。事务提交后，后台任务再把这条事件投递到 MQ。  
这样能解决“订单写成功了，但消息没发出去”的问题。  
它的关键价值是：把“数据库写入”和“事件产生”绑在一个本地事务里。
2. 可靠重试  
消费者收到消息后处理失败，不是直接放弃，而是按规则重试。  
常见做法是：固定重试、指数退避、延迟队列、最大重试次数、失败后进死信队列。  
这适合处理临时性故障，比如网络抖动、下游超时、锁冲突。  
前提是消费逻辑必须幂等，否则重试会把业务做重。
3. 补偿事务  
如果某一步已经成功、后续步骤失败，就执行“反向操作”把系统拉回合理状态。  
比如订单已创建，但库存扣减失败，那就把订单状态改成取消；  
或者支付成功后发券失败，就补发券，实在不行就退款。  
补偿事务不是回滚数据库，而是用业务语义“纠偏”。
4. 定时对账 / 巡检修复  
这是最终兜底。  
即使 MQ 丢了、重试漏了、补偿没跑到，也可以通过定时任务扫描异常状态，把脏数据补回来。  
比如每天扫“已支付但订单未完成”的记录，重新触发履约；  
或者扫“outbox 未投递成功”的事件，重新发送。  
它保证系统不是“出错就永远错下去”。  


如果面试官继续追问，你再补一句：  
“像积分、通知、履约这类链路我会优先用最终一致；资金扣减这类强一致要求高的场景，会更谨慎，尽量缩小异步范围。”
