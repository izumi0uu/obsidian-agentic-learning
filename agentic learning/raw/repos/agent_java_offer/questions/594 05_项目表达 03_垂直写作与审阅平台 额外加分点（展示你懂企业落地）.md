### 额外加分点（展示你懂企业落地）

- **可扩展**：支持多租户（不同医院独立实例）。
- **[[Evaluation|评估]]指标**：临床准确率（与医生一致性>95%）、幻觉率<1%、处理速度、ROI（用户节省时间）。
- **风险控制**：Agent只做“建议+证据链”；所有输出带来源溯源。





## 3. 怎么降低“假引用”和幻觉？

这是高频题，你可以这么答：

“我会从四层做控制。  
第一层是数据源控制：只允许白名单源进入关键链路，比如 PubMed、官方指南、临床试验注册库、专利库。  
第二层是生成前约束：先抽证据再生成，要求引用 ID 先被检索系统验证。  
第三层是生成后校验：检查引用是否存在、标题是否匹配、结论是否能被原文支撑。  
第四层是结果展示约束：把‘事实’和‘模型推断’分开展示，对不确定内容明确标注。”


**参考答案：**

1. **降低幻觉：** RAG 通过提供事实依据，让回答有据可循。微调虽然能注入知识，但模型仍可能在不确定时产生幻觉。
2. **衡量“事实性/幻觉水平”：**
  - **基于知识库的 QA：** 构建一个问题集，其中每个问题的答案都可以从一个确定的知识源中找到。
  - **对抗性问题：** 设计一些诱导模型产生幻觉的问题，比如询问关于不存在的人物或事件的信息。
  - **评估方法：** 使用 LLM-as-a-Judge 或 Faithfulness 指标，判断生成答案是否与提供的源知识相符。

## 4. 如果让你做评估体系，你怎么评？

你答：

“我会把评估拆成四层：  
一是检索质量，看召回是否全、排序是否准、来源是否权威；  
二是事实性，看结论和引用的一致性、引用真实性、是否越界推理；  
三是任务完成度，看有没有真正回答用户的问题，而不是只给摘要；  
四是用户价值，看节省了多少时间、是否能直接进入下游工作流。  
如果是生产环境，我会做离线评估 + 人工专家抽检 + 线上行为指标结合。”


**参考答案：**

全面地评估一个 RAG 系统，必须将其拆分为**检索阶段**和**生成阶段**两个独立但又相互关联的部分进行评估。

**第一阶段：检索性能评估（Retrieval Evaluation）**

这个阶段的目标是评估检索器能否“找得对、找得全”。

1. **上下文精确率（Context Precision）：** 衡量检索到的文档中有多少是真正与问题相关的。
2. **上下文召回率（Context Recall）：** 衡量所有相关的文档中，有多少被检索器成功找回来了。
3. **Hit Rate：** 检索到的文档中是否至少包含一个相关文档。
4. **MRR：** 第一个相关文档排名的倒数的平均值。
5. **nDCG@k：** 同时考虑检索结果的相关性等级和它们在结果列表中的排名。

**第二阶段：生成性能评估（Generation Evaluation）**

这个阶段的目标是评估 LLM 在给定上下文后，能否生成“忠实、准确、有用”的答案。

1. **忠实度/可溯源性（Faithfulness / Groundedness）：** 生成的答案是否完全基于所提供的上下文，是否存在捏造或幻觉。
2. **答案相关性（Answer Relevancy）：** 生成的答案是否直接、清晰地回答了用户的原始问题。
3. **答案正确性（Answer Correctness）：** 答案中的信息是否事实准确。

另外，对已上线的系统还要持续监控输入分布、输出质量、用户反馈和自动化评估结果，以应对数据漂移和模型漂移。

## 5. Workflow 和 Agent，你会怎么选？

参考答法：

“我一般会从任务的稳定性和开放度来选。  
如果任务路径相对固定，比如文献检索、摘要生成、引用整理、格式输出，这类场景我会优先选 workflow，因为更稳定、可控、好评估。  
如果任务比较开放，用户目标不明确，需要系统自己拆步骤、决定调用哪些工具，那我会考虑 Agent。  
所以我不会把 Agent 当成默认答案。生产环境里经常是 workflow 为主，Agent 为辅，用 Agent 去补 workflow 覆盖不到的部分。”

## 7. 你会怎么降低幻觉？

参考答法：

“我会从系统层做，而不是只依赖 prompt。  
第一层是数据源控制，只允许高可信来源进入关键链路。  
第二层是检索增强，让模型尽量基于真实材料回答，而不是凭参数记忆。  
第三层是生成约束，比如要求结构化输出、强制引用、区分事实和推断。  
第四层是结果校验，检查引用真实性、标题匹配度、结论是否被证据支持。  
如果场景再严一点，我还会加人工复核或者专家抽检。  
我的理解是，降低幻觉本质上是一个系统设计问题，而不是一个 prompt 小技巧。”


**参考答案：**

1. RAG 通过提供具体的、相关的上下文，将 LLM 的回答“锚定”在这些事实依据上，显著降低了幻觉的产生。
2. RAG 可以清晰地展示出答案是基于哪些源文档生成的，用户可以点击查看来源进行事实核查。
3. **如何衡量“事实性/幻觉水平”：**
  - **基于知识库的 QA：** 构建一个问题集，其中每个问题的答案都可以从一个确定的知识源（如 Wikipedia、公司内部文档、数据库）中找到。
  - **对抗性问题：** 设计一些诱导模型产生幻觉的问题，比如询问关于不存在的人物或事件的信息。
  - **评估方法：** 使用一个更强大的 LLM，让它判断生成的答案是否与提供的源知识相符或矛盾。

## 8. 如果用户上传很多文档，你会怎么做知识库检索？

参考答法：

我会先做文档预处理，包括清洗、切分、元数据抽取和索引构建。  
切分时不会只做机械 chunk，我会尽量结合段落、标题、语义边界来处理，避免上下文断裂。  
然后是 embedding 建索引，召回时可以先做向量检索，再加关键词召回，最后用 rerank 提高相关性。  
拿到检索结果后，再按时间、来源、主题去做二次筛选，把最相关的内容拼到上下文里。  
如果要让用户信任结果，我还会把引用位置、原文片段和来源一起返回。


**参考答案：**

一个完整的 RAG 流水线可以分为两个主要阶段：**离线的数据准备（索引）阶段**和**在线的查询（推理）阶段**。

**离线阶段：**

1. 数据加载（Load）：从各种数据源加载原始文档。
2. 文本切块（Split / Chunk）：将长文档切割成更小的、语义完整的文本块。
3. 嵌入（Embed）：使用嵌入模型将每一个文本块转换成高维向量。
4. 存储（Store）：将文本块内容及其向量存储到向量数据库。

**在线阶段：**

1. 用户提问（User Query）
2. 查询嵌入（Embed Query）
3. 向量检索（Retrieve）
4. （可选）重排序（Re-rank）
5. 增强与生成（Augment & Generate）

文本切块策略没有唯一的最佳方案。实践中通常会从一个合理基线开始，然后通过评估检索质量，针对具体文档类型和查询场景进行迭代优化。有时甚至会采用多尺度切块策略，即同时索引不同大小的块，以应对不同粒度的查询。

基础的向量检索之外，还可以使用混合搜索（BM25 + 向量检索）、重排序（Cross-Encoder）、多查询检索（Multi-Query Retrieval）等技术来提升检索质量。

## 9. 如果让你用 Python 设计一个 AI 服务后端，你会怎么搭？

参考答法：

“我会优先用 FastAPI 来搭服务层，因为它比较适合做 AI 应用接口，开发效率高，也方便做异步。  
数据层可以根据场景接 MySQL 或 Postgres，缓存和会话状态可以用 Redis。  
如果有长任务，比如检索、生成报告、处理文件，我会引入异步任务机制，比如任务队列或者后台任务。  
模型调用层我会单独封装，做统一的请求、重试、日志、限流和 fallback。  
如果前端需要流式输出，我会提供 SSE 或 WebSocket。  
另外我会比较关注可观测性，比如日志、trace、错误监控和 token 成本统计，因为 AI 服务上线后这些都很重要。”


**参考答案：**

前后端采用 React + FastAPI + LangChain + Chroma + SQLite + SQLModel 架构。

我会优先用 FastAPI 来搭服务层，因为它比较适合做 AI 应用接口，开发效率高，也方便做异步。数据层可以根据场景接 MySQL 或 Postgres，缓存和会话状态可以用 Redis。如果有长任务，比如检索、生成报告、处理文件，我会引入异步任务机制，比如任务队列或者后台任务。模型调用层我会单独封装，做统一的请求、重试、日志、限流和 fallback。如果前端需要流式输出，我会提供 SSE 或 WebSocket。

我用分层和契约两条线保证清晰度。分层上，API 层只处理入参校验和响应协议；Service 层负责业务编排和规则决策；Repository/Model 层只负责数据读写与实体映射；Workflow 层专注状态转移与节点执行。契约上，层间统一使用 DTO 和明确的错误码，不透传底层实现细节。这样即使替换 LLM、切换向量库或升级存储，也主要影响局部层，不会牵动整条链路，代码可维护性和可测试性都更高。

## 10. 如果用户问一个很复杂的问题，Agent 怎么拆解任务？

参考答法：

“我会先让系统识别任务意图和最终目标，再把它拆成几个可执行的子问题。  
每个子问题再匹配最适合的工具，比如检索、数据库查询、摘要、对比分析、格式化输出。  
执行时我会尽量保留中间结果，避免模型每一步都从零开始想。  
等所有子任务完成后，再把中间结果做聚合，统一生成最终答案。  
同时我会设置边界，比如最大步骤数、工具调用白名单和失败回退逻辑，避免 Agent 无限制发散。”

## 11. 你怎么理解 ToB 和 ToC 的差异？如果公司从 ToB 往 ToC 做，你觉得技术上会有什么变化？

参考答法：

“ToB 和 ToC 的重点不太一样。  
ToB 更看重专业能力、准确性、定制化和交付深度，很多时候功能不一定花哨，但要稳定、可信、能融入客户流程。  
ToC 更看重体验、易用性、响应速度、留存和增长。  
如果公司从 ToB 往 ToC 做，我觉得技术上会更强调几个点：  
第一，交互体验要更轻更顺滑；  
第二，系统并发和成本控制会更敏感；  
第三，用户路径要更短，不能太依赖复杂配置；  
第四，评估指标会从交付成功扩展到留存、活跃和转化。  
所以技术架构上可能要兼顾快速试错和规模化能力。”



Q1：为什么选择LangGraph而不是CrewAI或AutoGen来编排医疗多Agent？
回答要点：
LangGraph是状态机（State Machine），每步状态（Task State）持久化到数据库，支持可回滚、可审计——医疗场景下必须满足HIPAA/NMPA审计要求。
CrewAI适合快速原型，但状态管理弱；LangGraph支持循环边（cycles）+ Human-in-the-Loop，完美匹配我们第6步（Critic & Verifier）和第8步（Auditor）。

Q2：如果某个Agent（比如Literature Retrieval）失败了，你的Workflow怎么处理？
回答要点：
用LangGraph的conditional edges + retry机制（内置exponential backoff）。
第1步Gatekeeper Agent先做风险评估；失败时自动路由到Human-in-the-Loop节点，同时记录完整日志（Prompt + Output + Error）。
医疗特有：如果检索不到最新指南，系统会标记“证据不足”

Q3：如何保证整个流程严格遵守HIPAA/GDPR/NMPA，尤其是PHI处理？
回答要点：
Tokenization + 最小化原则：Gatekeeper Agent（第1步）在入口就把患者姓名/MRN脱敏成Token，只有必要Agent能看到原始数据，且所有Agent运行在私有化/零保留（Zero-Retention）环境。
每步输出都带audit trail（LangGraph的checkpointer自动保存），支持事后追溯。

Q5：医疗场景下如何有效降低LLM幻觉？你们Workflow里具体用了哪些技术？
回答要点：
多Agent协作验证：第5步Writer生成后，第6步Critic Agent模拟“同行评审”（Cross-check文献、GRADE证据等级）。
技术栈：RAG（实时抓PubMed）+ Tool Calling（调用真实数据库）+ Reflection Loop（让Agent自己反思3次）。
指标：目标幻觉率<1%，通过Human + Automated Eval双重把关。

Q8：如果输入包含医学影像或多模态数据（CT/MRI + 检验单），Workflow如何扩展？
回答要点：
在第3步增加Vision Agent（CheXagent类模型），输出结构化影像报告后合并到主流程。
LangGraph支持多模态状态，无缝对接。

Q12：你如何持续迭代优化这个Workflow？
回答要点：
每周收集医生Feedback → 自动触发RAG更新 + Agent Prompt优化。
用LangSmith/LangGraph监控每个Agent的Latency & Success Rate，自动告警。
