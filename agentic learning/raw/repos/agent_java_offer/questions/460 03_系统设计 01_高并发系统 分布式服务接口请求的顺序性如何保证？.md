## 题目正文

### 分布式服务接口请求的顺序性如何保证？

首先，工程上更推荐的思路是：**尽量从业务设计上避免强顺序依赖**。因为一旦引入严格顺序保证，通常会带来：
- 系统复杂度上升；
- 吞吐下降；
- 热点 key 压力变大；
- 排障和恢复都更麻烦。

如果业务确实要求顺序性，常见解法有三类：

1. **Hash 分发到同一机器**
- 比如按订单 ID 做一致性 hash，把同一业务对象的请求路由到同一台机器；
- 到了该机器内部，再把同一 key 的请求扔到内存队列里串行处理；
- 这样能做到“同一 key 有序，不同 key 并行”。

2. **MQ / 内存队列串行化**
- 生产端按业务 key 发到同一个 partition 或同一条队列；
- 消费端再对 key 做 hash，落到固定线程或固定内存队列里顺序消费。

3. **锁与序列号控制**
- 可以通过分布式锁、数据库行锁，或者请求携带版本号 / 序列号来控制乱序更新；
- 但这种做法更重，吞吐和热点问题会更明显。

如果是更新型请求，还可以在各阶段加幂等保护，比如：
- 执行前先检查 redis 幂等 key；
- 执行过程中对异常做回滚和恢复；
- 成功后再修改状态位，避免重复执行。

面试里可以总结成一句：
**顺序性不是白拿的，核心思路是“同一 key 聚合到同一路径上串行处理”，常见手段是 hash 路由 + 队列串行 + 幂等兜底。**
