### 原文内容

### 微信朋友圈这样的社交系统，应该如何设计？

一个比较通俗的拆法是：
1. **发布链路**
- 用户点击发朋友圈时，客户端先把内容缓存到本地，立刻给用户成功反馈；
- 图片/视频先就近上传到 CDN；
- 客户端把图片地址、文字、可见范围等元数据提交给后端；
- 后端写入“朋友圈发布表”。

2. **索引层 / 相册层**
- 发布表写成功后，再往相册表写一条索引，指向真正的发布内容；
- 这样用户查看自己历史内容时，可以先走索引，再回源到发布表。

3. **时间线分发**
- 后台离线任务或异步任务，根据可见范围，把这条朋友圈写入好友的时间线表；
- 好友刷朋友圈时，本质上查的是时间线表，而不是全局扫发布表。

4. **读取链路**
- 时间线里拿到帖子索引；
- 再根据图片 URL 去 CDN 拉内容；
- 有新内容时可做红点提醒。

### 朋友圈权限控制怎么做？

权限本质上是“关系 + 可见范围 + 用户配置”的联合判断。

一种可讲的工程化思路是：
- 用户和好友之间的关系、朋友圈权限设置、总可见范围这些信息，都需要在服务端持久化；
- 同时客户端会缓存一部分权限数据，用于本地快速判断和展示优化；
- 每次刷新朋友圈时，再结合当前用户与发帖人的关系、帖子本身的可见范围，做一次最终判定。

一句话说：
**发布时控制可分发范围，读取时再做一次权限校验，形成双层控制。**

### 如何设计高并发的朋友圈点赞系统架构？

点赞和评论通常是独立数据，不建议和主帖子强耦合。

常见做法：
1. **Redis 存点赞集合**
- 每条朋友圈对应一个 Redis Set；
- Set 里放给这条朋友圈点赞的用户 ID；
- 点赞数量可以直接用 `SCARD`，点赞列表可用 `SMEMBERS` 或分页拉取。

2. **好友可见点赞**
- 如果“只有好友才能看见某个点赞”，可以把“当前用户的好友集合”和“这条朋友圈的点赞用户集合”做交集判断；
- 工程上可以通过 `SISMEMBER` 之类方式快速判断是否可见。

3. **重复点赞幂等**
- 如果用数据库存点赞关系，可以用 `(user_id, post_id)` 联合唯一键保证一个人只能点一次；
- 如果用 Redis Set，本身天然去重，重复添加不会产生第二条记录。

面试里可以总结为：
**朋友圈主链路是“发帖异步分发到时间线”，点赞链路是“独立存储、缓存优先、关系裁剪可见性、幂等防重”。**
