## 2. 补充材料：为什么需要执行沙箱

### 4) Understanding Manus sandbox - your cloud computer

- 原文链接：
  - [https://manus.im/blog/manus-sandbox](https://manus.im/blog/manus-sandbox)
- 核心思想：
  - 将 Agent 执行环境放在沙箱中，形成可隔离、可复现、可回放的“云电脑”执行面。
  - 沙箱让[[Tool Calling|工具调用]]不直接触达核心系统，降低高权限动作风险。
  - 文章强调“思考能力”必须和“安全执行环境”配套，才能上线生产场景。
- 文章概述（约500~1000字）：
  - 这篇文章的重点是把 Agent 的“执行面”讲清楚：模型会推理并不等于系统可以安全上线，真正落地还需要一个可控的运行环境。Manus 把这个运行环境定义为 sandbox（云电脑），核心价值是隔离、可复现、可治理。隔离意味着 Agent 的浏览器操作、代码执行、文件读写都在受控环境中完成，不直接接触用户本机或核心生产系统；可复现意味着每次任务过程可追踪、可回放，便于故障排查和合规审计；可治理意味着可以施加权限边界、资源配额、网络策略和操作白名单。文章强调很多 Agent 事故并非“模型胡说”，而是执行环节缺乏约束，例如重复提交、高权限误操作、异常重试放大。沙箱机制相当于给 Agent 套上工程安全带，把高风险动作从“默认放行”改成“可验证后执行”。从面试表达看，你可以把这篇文章提炼为三层：安全层（权限/隔离）、稳定层（重试/回滚/断点恢复）、观测层（日志/[[Trace|轨迹]]/审计）。再补一句“Agent 的生产门槛不是会不会回答，而是能不能安全执行”，就能把你和只谈模型能力的候选人拉开差距。
- 面试可能问的点：
  - 问：为什么 Agent 项目必须做执行沙箱？
  答：因为 Agent 一旦具备网页操作、代码执行和外部调用能力，风险从“答错”升级为“误执行”。沙箱把执行面与核心系统隔离，能限制破坏范围，避免高权限误操作直达生产环境。没有沙箱，很多企业场景即使效果好也不敢上线。
  - 问：沙箱里如何做权限控制、资源限制和审计回放？
  答：我会做[[Least Privilege Tools|最小权限]]白名单（可访问域名、可用工具、可写目录），再加 CPU/内存/时长限额和网络出口策略。所有关键动作记录为结构化事件日志，绑定任务ID和操作者，支持按步骤回放，便于定位问题、追责和合规审计。
  - 问：沙箱模式对稳定性、排障和合规有什么收益？
  答：稳定性上可通过隔离和限额避免故障扩散；排障上有可回放轨迹，能快速复现实例并修复；合规上可形成“谁在何时做了什么”的审计链。对企业来说，这三点比单次模型效果更关键，决定系统是否可持续运营。
