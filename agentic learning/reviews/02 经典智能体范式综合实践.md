# 02 经典智能体范式综合实践

日期：2026-05-12

## 来源

这页承接一组章节综合题，主题是三种经典智能体范式：[[ReAct]]、[[Plan-and-Solve Prompting]] 和 Reflection / [[Reflexion]]。

边界：这里的 `Reflection` 泛指“执行-反思-优化”机制；[[Reflexion]] 是更具体的论文范式，强调 `Trajectory -> Evaluator -> Self-reflection -> Experience -> Actor`。答题时可以借用 [[Reflexion]]，但不要把所有 reflection 机制都等同于这篇论文。

## 使用规则

- 从 `02-A` 开始，每次只答一个小节或一个实践任务。
- 先写自己的答案，再让我补反馈。
- 代码实践题先记录假设、工具接口、输入输出，再写代码。
- 每个小节答完后，由 Codex 判断是否写回 [[Agent Workflow]]、[[Agent Harness]]、[[Tool Calling]]、[[Memory Reflection]]、[[Evaluation]]、[[05 Query 写回队列]] 或概念对比页。
- 本页是学习过程记录，不是 raw evidence，也不直接替代概念卡。

## 当前目标

把“听过三种范式”推进到“能在具体系统里选择、组合、实现、识别风险”。

重点不是背定义，而是能回答：

- 哪个机制介入行动前、行动中、行动后？
- 哪些能力属于 LLM 输出，哪些属于框架 / harness / workflow？
- 工具、解析、状态、重试、评估和人类审批如何让范式变成工程系统？

## 总收束标准

这页不要求一次答完。完成后应能解释：

- ReAct、Plan-and-Solve、Reflection 分别在行动前、行动中、行动后解决什么问题。
- 为什么生产系统通常不能只靠正则解析 LLM 输出。
- 工具数量变多后，为什么需要 registry、retrieval、permissioning 和 routing。
- 动态重规划为什么依赖 state、observation、evaluator 和 replan 节点。
- Reflection 为什么需要更智能的终止条件，而不是只靠字符串判断。
- 客服智能体为什么必须把自动化收益和权限、合规、用户体验一起设计。
