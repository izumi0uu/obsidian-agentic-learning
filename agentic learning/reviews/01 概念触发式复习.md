# 01 概念触发式复习

日期：2026-05-10

这页记录概念触发式复习：先保留我的原始解释，再由 Codex 给出校准版、边界提醒和追问。它不是 raw evidence，也不是最终概念卡。

## 复习节奏规则

- 默认只做一轮追问：用来暴露主要误解和边界缺口。
- 如果第一轮仍有关键卡点，最多追加一轮第二轮追问；第二轮问题应聚焦 1-3 个最影响理解的缺口，不再铺开新战线。
- 第二轮之后必须收束：只做总结、写回候选、补概念卡或加入 [[02 问题池]] / [[05 Query 写回队列]]，不再生成第三轮追问。
- 如果还没懂，说明材料或概念卡需要重写，而不是继续加问。

# ReAct

## 目标

用费曼方式检查我是否真的理解 [[ReAct]]：不是背定义，而是能说清它的执行分工、循环边界和现代工程落点。

## 我的原始解释

> ReAct，我会解释它是什么，它是 LLM 的一种 Agent loop 范式，LLM 每轮输出下一步想法/动作，外部框架负责执行动作（tool calling），拿到 observation（tool 返回结果），塞到上下文，然后 LLM 再输出想法或动作，直到想法已经觉得解决了问题。

## 校准版

ReAct 是 Reasoning + Acting 的 [[Agent Loop]] 范式。模型每轮根据上下文输出下一步推理意图和动作请求；外部框架解析或接收这个 action，通过 [[Tool Calling]] 执行真实工具，拿到 [[Observation]]，再把 observation 写回上下文或显式 state。模型基于新的 observation 继续决定下一步，直到输出 final answer，或者运行框架因为成功条件、最大步数、错误策略、预算上限或人工接管而停止。

## 我已经说对的点

- 抓住了“LLM 每轮只输出下一步”。
- 抓住了“外部框架负责执行工具和驱动循环”。
- 抓住了 “observation 回填上下文后再进入下一轮”。

## 需要更精确的点

- ReAct 不是 LLM 自己在后台循环，loop controller 在框架或 [[Agent Harness]] 里。
- `Action` 可以用现代 [[Tool Calling]] 接管，不一定继续依赖纯文本 `Action: ...` 格式。
- [[Observation]] 不只等于 tool 返回值，也可以是环境状态、网页状态、代码执行结果、测试反馈或用户反馈。
- 停止条件不只由“模型觉得解决了”决定，还可以由框架的成功条件、最大步数、预算、错误策略或人工审批决定。

## 第一轮追问

1. 如果 LLM 每轮只输出文本，为什么说 ReAct 的循环不是 LLM 自己跑起来的？
2. ReAct 和普通一次性 tool calling 的最小区别是什么？
3. Observation 和上下文、Agent State、Memory 的边界分别是什么？
4. 为什么生产系统通常不会只靠裸 `Thought -> Action -> Observation` prompt 跑 ReAct？
5. 如果 ReAct 陷入原地打转，你会从框架层加哪些停止或纠错机制？

## 我的费曼回答区

### Q1：如果 LLM 每轮只输出文本，为什么说 ReAct 的循环不是 LLM 自己跑起来的？

我的回答：

LLM是大语言模型，它职责是根据上下文去输出文本，本质是概率模型，根据前一个字猜下一个字。所以现在的ReAct循环是由agent框架（代码）去调控的，LLM本身不具备循环能力，所以正确循环是，训练了ReAct的LLM会返回actions，stop等等文本，代码会判断而去做操作。在概念一开始时，是以`Thought -> Action -> Observation` prompt，作为LLM上下文去让LLM主观返回。

反馈：

这个回答抓住了核心：LLM 本身只负责根据上下文生成下一段输出，循环控制、工具执行、Observation 回填和停止判断都在 [[Agent Harness]] / 框架里。

需要再精确一点：不是只有“训练了 ReAct 的 LLM”才会返回 action。模型可以通过 prompt、few-shot 示例、fine-tuning、system instruction 或现代 [[Tool Calling]] schema 被引导输出 action / tool call。`stop` 也不一定是模型返回的文本；==很多时候是框架根据 final answer、最大步数、预算、错误次数、测试结果或人工确认来停。==

写回：

- 已掌握：ReAct loop controller 在框架，不在 LLM 内部。暂不需要新增写回；[[ReAct]] 已覆盖这个边界。

### Q2：ReAct 和普通一次性 tool calling 的最小区别是什么？

我的回答：

它是agent loop，是可循环的流程，是会有多次+LLM判断思考的。

反馈：

方向对，但还需要补一个关键词：==**Observation 驱动下一轮决策**。==

普通一次性 tool calling 通常是：

```text
用户问题 -> LLM 生成一次 tool call -> 工具返回 -> LLM 汇总答案
```

ReAct 更像：

```text
LLM 决定下一步 action -> 工具/环境返回 observation -> observation 进入上下文或 state -> LLM 再决定下一步 action 或 final answer
```

所以最小区别不是“调用工具次数一定更多”，而是工具结果会不会作为 [[Observation]] 进入下一轮决策循环。

写回：

- 候选：把“一次性 tool calling vs ReAct = 是否由 Observation 驱动下一轮决策”补进 [[ReAct]] 或 [[Tool Calling]] 的边界段。

### Q3：Observation 和上下文、Agent State、Memory 的边界分别是什么？

我的回答：

Observation是LLM每次对话或每次agent loop中对外界的观察，比如tool调用的结果，接口的返回。上下文是一次会话窗口中LLM的接受信息的总和，agent state是当前任务工作状态，是短期的，包括，目标，当前阶段，计划，已完成步骤，工具返回结果，错误，等待人工确认，下一步依据等等，meory是跨任务，跨对话的信息，保持用户喜好，历史事实，长期经验等等。

反馈：

这题回答得很好，四个概念基本切开了。

可以再压成一句：

```text
Observation = 外部这一步返回了什么
Context = 本轮模型能看到什么
Agent State = 当前任务运行到哪里
Memory = 以后还能复用什么
```

再补一个细节：Observation 不一定直接等于 context。它可能先进入 [[Agent State]] 或 [[Trace]]，再由框架选择其中一部分塞进 context。Memory 也一样，不是所有长期记忆都会每轮进入上下文。

写回：

- 已掌握：Observation / Context / Agent State / Memory 的边界清楚。可作为后续复习的标准答案。

### Q4：为什么生产系统通常不会只靠裸 `Thought -> Action -> Observation` prompt 跑 ReAct？

我的回答：

因为这样不确定性会很大，完全会因为上下文限制而丢失`Thought -> Action -> Observation`中的信息，所以现代会以agent框架去用ReAct范式去实现。

反馈：

这个回答方向对，但原因不只是==上下文限制==。生产系统不用裸 prompt 跑 ReAct，主要因为裸文本 loop 太脆：

- ==action 格式可能不稳定，解析失败。==
- ==工具参数可能缺字段、类型错或越权。==
- ==工具执行前缺少权限、审批和安全检查。==
- ==Observation 可能污染上下文或被模型误读。==
- ==没有稳定的 retry、timeout、最大步数、预算和停止条件。==
- ==trace / eval / replay 不充分，出错后难复现。==

所以现代系统会用 [[Tool Calling]] schema、[[Agent State]]、[[Agent Lifecycle Hook]]、[[Guardrails]]、[[Trace]]、[[Evaluation]] 和 [[Human-in-the-loop]] 接管这些边界。

写回：

- 候选：把“裸 ReAct prompt 的脆弱点清单”保留为 [[ReAct]] 的复习材料；概念卡已有现代系统吸收段，暂不必重复写入。

### Q5：如果 ReAct 陷入原地打转，你会从框架层加哪些停止或纠错机制？

我的回答：

tool的介入。

反馈：

这题目前回答还太薄。“tool 的介入”只是一个方向，但不够说明框架层怎么止损。

==更完整的框架层机制可以分几类：==

```text
停止：最大步数、最大工具调用次数、预算上限、超时、连续相同 action 检测
纠错：工具错误重试、参数修正、query rewrite、切换工具、重新规划
评估：evaluator 检查是否接近目标、trajectory evaluation、测试/断言
升级：低置信度时 ask user、human approval、handoff 给另一个 agent
记账：把失败路径写进 trace/state，避免下一轮重复同一条路
```

如果只说“tool 介入”，容易误以为多调用工具就能解决循环。真正关键是：框架要能判断“这一步没有带来新信息”，然后停止、改路或升级。

写回：

- 待补强：下一轮重点复述“停止条件 / 纠错机制 / 评估机制 / 人工升级”四类。
## 写回候选

- [x] Q1 已能清楚解释“loop controller 在框架里”；[[ReAct]] 已覆盖。
- [x] Q3 已能切开 Observation / Context / Agent State / Memory；暂不加入问题池。
- [ ] Q2 候选写回：一次性 tool calling vs ReAct 的最小区别是“是否由 Observation 驱动下一轮决策”。
- [ ] Q5 继续复习：把停止条件、纠错机制、评估机制、人工升级四类说完整后，再考虑写回 [[Agent Harness]] 或 [[Agent Framework]]。

## 第二轮追问

1. 请用一句话区分“一次性 tool calling”和 ReAct，必须出现 `Observation`。
2. 如果连续两轮 action 使用同一个工具、同一组参数、得到相似 observation，框架应该怎么判断和处理？
3. 为什么“多加工具”不一定能解决 ReAct 原地打转？
4. ReAct 的停止条件里，哪些可以由模型判断，哪些最好由框架判断？
5. 如果你要设计一个最小 ReAct harness，你会放哪 5 个字段进 [[Agent State]]？

## 第二轮收束标准

当我能不用看卡片解释这句话时，本轮 ReAct 复习收束；如果仍卡住，只把缺口写入 [[02 问题池]]，不再追加第三轮追问：

> ReAct 的核心不是固定的 prompt 格式，而是“模型提出下一步，框架执行动作并回填观察，再让模型基于观察继续决策”的行动循环分工。

# Plan-and-Solve Prompting

日期：2026-05-10

## 目标

用费曼方式检查我是否能区分 [[Plan-and-Solve Prompting]]、[[Planning]] 和现代 [[Agent Workflow]]：尤其是“先计划再执行”什么时候只是 prompt 方法，什么时候已经进入带 tool / observation / replan 的 Agent workflow。

## 我的原始解释

> 我会解释 planing and solve 概念，它同样是一个 agent loop 范式，其特点在于，用户发送 request 给 agent 后，agent 会先 plan 一些解决问题的 tasks，然后 agent 会 loop 统一执行，然后会 replan 是否重新 plan 一些 tasks，还是判断已经解决了问题，最终把答案发给用户，常见于一些可以被清晰拆解的负责任务，比如数学问题，代码编写任务等等。

## 校准版

如果严格说论文里的 [[Plan-and-Solve Prompting]]，它不是 Agent loop 范式，而是 zero-shot CoT 的一种 prompting 方法：用户给出问题后，模型先生成一个 plan，再按 plan 求解，最后输出答案。这个过程通常发生在一次模型回答里，没有外部 tool action，也没有 [[Observation]] 回填，更没有运行时层面的 replan。

你描述的“先拆 tasks -> 执行 loop -> 根据结果 replan -> 判断是否完成”更接近现代 Agent 里的 plan-and-execute / replan [[Agent Workflow]]。这种 workflow 会把 plan 变成任务列表或显式 state，再由框架驱动执行、观察、评估和重规划。代码编写任务很适合这种工程形态；数学题如果只是纸面推理，更常见的是 Plan-and-Solve Prompting，而不是完整 Agent workflow。

## 我已经说对的点

- 抓住了“先 plan，再 solve / execute”的核心直觉。
- 抓住了复杂任务适合先拆解的场景，尤其是代码编写、系统性问题、可分解任务。
- 抓住了现代 Agent workflow 中会出现 replan 和完成判断。

## 需要更精确的点

- [[Plan-and-Solve Prompting]] 本体不是 Agent loop；它没有外部 action / observation。
- “tasks + loop + replan” 是 plan-and-execute [[Agent Workflow]]，不是论文里的 Plan-and-Solve Prompting。
- 数学题是 Plan-and-Solve 的经典适用场景，因为它可以在文本推理里先列步骤再求解；代码编写任务通常会升级成 Agent workflow，因为它需要读文件、改代码、跑测试和处理 observation。
- 停止条件在 prompt 方法里多由模型输出答案完成；在 Agent workflow 里则通常由框架的测试、验收标准、预算、最大步数、人工确认或 evaluator 决定。

## 第一轮追问

1. 如果没有工具调用和 observation，为什么 Plan-and-Solve Prompting 还不能算完整 Agent loop？
2. “先写 plan” 和“把 plan 放进显式 Agent State 并执行”差别在哪里？
3. 数学题为什么更像 Plan-and-Solve，而代码任务为什么更容易变成 Agent Workflow？
4. ReAct 和 plan-and-execute workflow 都有 loop，它们的 loop 触发条件有什么不同？
5. 如果 plan 一开始就错了，prompt-only 的 Plan-and-Solve 和框架化 Agent Workflow 分别能怎么补救？

## 我的费曼回答区

### Q1：如果没有工具调用和 observation，为什么 Plan-and-Solve Prompting 还不能算完整 Agent loop？

我的回答：

缺少了这两个当然不算agent loop，Plan-and-Solve Prompting 是基于prompt出发的，也就是预先让LLM生成一系列tasks，它自己循环去解决最终输出答案。

反馈：

方向对：你抓住了“没有工具调用 / 没有 observation，所以不能算完整 Agent loop”。

需要修正一个词：Plan-and-Solve Prompting 里最好不要说 LLM “自己循环”。更准确是：模型在一次回答内部先写 plan，再按 plan 生成 solve 过程和最终答案。这个过程可以有多步推理文本，但没有外部 runtime 一轮轮调用模型、执行工具、回填 observation，也没有框架层的 stop / retry / replan。

一句标准答案：

```text
Plan-and-Solve Prompting 是 prompt 层的 plan -> solve 文本推理，不是 runtime 层的 action -> observation 循环。
```

写回：

- [x] 已掌握核心边界：没有外部 action / observation / runtime controller，就不能算完整 Agent loop。

### Q2：“先写 plan” 和“把 plan 放进显式 Agent State 并执行”差别在哪里？

我的回答：

先写plan等于让LLM先根据plan第一步就做plan的推理，把 plan 放进显式 Agent State 并执行，相当于是在中途插入上下文，所以LLM会在下一步进行plan的推理。

反馈：

这题是本轮最需要补强的地方。你把 Agent State 理解成“中途插入上下文”，这个只说到了一小部分。

更准确的边界是：

```text
plan 文本 = 模型输出里的一段推理/任务草稿
Agent State 里的 plan = runtime 可读、可更新、可检查的任务状态
context = 当前这一轮模型实际看见的信息投影
```

把 plan 放进 [[Agent State]] 不是简单塞回上下文，而是让框架能管理它：每个任务是否完成、依赖谁、用了哪些工具、失败几次、下一步该执行哪个节点、是否需要 replan、是否需要人工确认。然后框架只把 state 中当前需要的部分投影进 context。

小边界：context 是“模型这轮能看到什么”；state 是“系统知道任务运行到哪里”。Plan-and-Solve Prompting 只有前者里的文本计划；Agent Workflow 会把计划变成后者里的执行对象。

写回：

- [ ] 仍需复习：`plan 文本 / Agent State / context 投影 / task list` 的边界。

### Q3：数学题为什么更像 Plan-and-Solve，而代码任务为什么更容易变成 Agent Workflow？

我的回答：

因为Plan-and-Solve易于解决复杂的问题，而数学题这种需要先plan好解决方案的类型题，是适合的，其实代码任务也是适合用Plan-and-Solve的，agent workflow有些也是遵循这种Plan-and-Solve模式。

反馈：

你说对了一半：数学题和代码任务都可以受益于“先 plan 再做”。但它们进入的层不一样。

数学题通常更像 [[Plan-and-Solve Prompting]]，因为问题、推理、计算和答案大多能在一次文本生成里完成。它需要显式步骤，但不一定需要外部工具、文件系统、测试或 observation 回填。

代码任务更容易升级成 [[Agent Workflow]]，因为真实代码任务经常需要读 repo、改文件、运行测试、看报错、再修复。这里的关键不是“有没有 plan”，而是有没有外部环境反馈：测试结果、lint 报错、文件 diff、编译失败，这些都是 observation，会推动下一轮执行或 replan。

写回：

- [x] 已掌握“代码任务也会吸收 Plan-and-Solve 思想”；需要记住：代码任务升级成 workflow 的原因是外部工具和 observation。

### Q4：ReAct 和 plan-and-execute workflow 都有 loop，它们的 loop 触发条件有什么不同？

我的回答：

plan-and-execute workflow的loop，触发条件是，先plan一份tasks，然后llm loop这份tasks，然后replan，决定是否继续追加tasks或返回最终输出；ReAct的loop是会根据observation来决定下一步是否loop，是属于LLM先思考后决定是否loop。

反馈：

这题回答得比较接近了，尤其是你抓住了 ReAct 由 observation 推动下一轮。

再精确一点：ReAct 的最小 loop 触发是“一次 action 之后拿到 observation，模型基于 observation 决定下一步 action 或 final answer”。Plan-and-execute workflow 的 loop 触发更多来自任务队列和执行评估：task 是否完成、结果是否通过测试、计划是否缺步骤、是否需要 replan、是否达到验收标准。

也就是说：

```text
ReAct loop：observation 驱动下一步动作选择
Plan-and-execute loop：task state / eval result 驱动继续执行或 replan
```

这里不要说“LLM loop 这份 tasks”。更准确是框架根据 task state 调度执行节点，LLM 可能参与执行、判断、总结或 replan，但 loop controller 仍在 workflow / harness。

写回：

- [x] 候选写回已能表达：ReAct 与 plan-and-execute workflow 的差异可以补进 [[Agent Workflow]] 的复习材料。

### Q5：如果 plan 一开始就错了，prompt-only 的 Plan-and-Solve 和框架化 Agent Workflow 分别能怎么补救？

我的回答：

prompt-only 的 Plan-and-Solve直接退出LLM思考，重新编辑prompt即可；框架化 Agent Workflow一般都是human in the loop，也可以中间插入新prompt对plan进行修改。

反馈：

方向上知道两者补救能力不同，但还可以更完整。

prompt-only 的 Plan-and-Solve 里，计划错了通常只能靠同一次输出里的自我修正、用户重新提示、或者再跑一次模型来补救。它没有稳定的外部检查点，也没有天然的工具结果来证明 plan 错在哪里。

框架化 Agent Workflow 的补救不只靠 human-in-the-loop。它可以有：


# Reflexion

日期：2026-05-11

## 目标

用费曼方式检查我是否真的理解 [[Reflexion]]：不是只记住“有反思和记忆”，而是能说清 `Trajectory -> Evaluator -> Self-reflection -> Experience -> Actor` 这条失败经验回流链路。

## 我的原始解释

> refexion是一个有LLM反思过程，长期记忆，短期记忆的一个机制。

## 校准版

[[Reflexion]] 是一种让语言 Agent 在一次尝试之后，根据执行轨迹或结果反馈生成反思文本，并把这段反思作为经验影响下一轮行动的机制。它的关键链路不是“有 memory”本身，而是：

```text
Trajectory -> Evaluator -> Self-reflection -> Experience -> Actor
```

其中 [[Trajectory]] 更像短期记忆：记录这次尝试中做了什么、观察到了什么、哪里失败或成功。Evaluator 把轨迹或最终结果转成反馈信号。Self-reflection 再把反馈写成自然语言经验。Experience 才是可被下一轮 Actor 读取的经验记忆，它可以进入上下文，也可能被系统保存成更长期的记忆。

所以 Reflexion 的重点是“失败或评价反馈如何变成下一轮可用的经验”，不是模型权重训练，也不是普通的自我解释。

## 我已经说对的点

- 抓住了 Reflexion 里有 LLM 生成反思文本。
- 抓住了它和短期轨迹、长期经验记忆有关。
- 抓住了它是一种机制，而不是单次提示词技巧。

## 需要更精确的点

- Reflexion 需要 evaluator / feedback；不是模型随便反省一句就算 Reflexion。
- 短期记忆更准确地说是 [[Trajectory]]：当前尝试的行动、观察和结果。
- 长期记忆更准确地说是 experience / reflective text：从反馈中提炼出的可复用经验。
- 反思文本必须被下一轮 Actor 使用，才形成 Reflexion loop。
- Reflexion 不修改模型权重，它是语言经验在上下文或记忆层的再利用。

## 第一轮追问

1. 请用一句话复述 `Trajectory -> Evaluator -> Self-reflection -> Experience -> Actor`，必须说清每一环负责什么。
2. Reflexion 和普通 [[Reasoning Trace]] 的区别是什么？
3. 如果 evaluator 给错反馈，Reflexion 会带来什么风险？
4. 为什么说 Reflexion 是“语言层的学习”，但不是模型权重训练？
5. 一条反思什么时候适合写入长期记忆，什么时候只适合留在当前任务上下文？

## 我的费曼回答区

### Q1：请用一句话复述 `Trajectory -> Evaluator -> Self-reflection -> Experience -> Actor`，必须说清每一环负责什么。

我的回答：

在执行过程中，每次action，agent都会选择性产生Trajectory这个短期记忆，记录这次action产生了什么效果，比如tool calling调用结果，然后记录，Evaluator去评估这次结果，是成功还是失败，Self-reflection是根据Evaluator的评估结果去让LLM再次反思自己的做法，产生Experience（长期记忆），LLM下次思考就可以使用，Actor就是LLM。

反馈：

这题回答得很接近了，主链路已经抓住：trajectory 记录发生了什么，evaluator 判断好坏，self-reflection 把反馈写成经验，actor 下一轮使用经验。

两个小边界再拧准：

- [[Trajectory]] 通常记录一次尝试或一段 episode 的行动轨迹，不一定是“每次 action 都单独产生一个短期记忆”。它可以包含 action、observation、工具结果、最终输出和成功/失败状态。
- Experience 不一定自动等于长期记忆。它先是“可被下一轮读取的反思经验”；只有当它可靠、可泛化、不会污染未来任务时，才适合沉淀为长期记忆。

可以压成标准句：

```text
Trajectory 记录本轮尝试；Evaluator 判断结果；Self-reflection 把反馈转成经验文本；Experience 被下一轮 Actor 读取，用来避免重复错误。
```

写回：

- [x] 已能复述 Reflexion 主链路；后续只需记住 Experience 不必然等于长期记忆。

### Q2：Reflexion 和普通 [[Reasoning Trace]] 的区别是什么？

我的回答：

Reasoning Trace是LLM在一次上下文输出文本（例如一次推理任务）中，LLM留下的显式推理步骤和过程。Reflexion是LLM具有反思评估再学习的过程，但是也只存在于agent或一次对话中，不属于训练范畴。

反馈：

核心区分是对的：[[Reasoning Trace]] 偏行动中 / 解题中的显式推理过程；Reflexion 偏行动后，根据反馈把经验写回下一轮。

需要修正一句：“只存在于 agent 或一次对话中”不够准确。Reflexion 确实不是权重训练，但 reflective text 可以只留在当前任务上下文，也可以被保存进 experience memory，在后续 trial 或类似任务中复用。关键不是它是否跨对话，而是它有没有形成“评价反馈 -> 反思经验 -> 下一轮使用”的闭环。

一句话边界：

```text
Reasoning Trace 记录模型怎么想；Reflexion 记录模型做完后从反馈里学到什么，并把它交给下一轮使用。
```

写回：

- [x] 已掌握 Reasoning Trace 与 Reflexion 的主边界。

### Q3：如果 evaluator 给错反馈，Reflexion 会带来什么风险？

我的回答：

LLM会逐步失控，输出无法解决问题的答案，偏离问题的输出。

反馈：

方向对：错误 evaluator 会让后续行动变差。但“逐步失控”可以再具体一点。

Reflexion 的风险是把错误反馈转成错误经验：比如本来工具参数错了，evaluator 却说“搜索方向错了”，self-reflection 就会生成错误教训。下一轮 Actor 读取这条经验后，可能重复错误策略、避开正确路径，或者把偶然失败泛化成错误规则。

如果这条错误 experience 被写入长期记忆，风险会扩大：它不只影响当前任务，还会污染后续相似任务。所以 Reflexion 必须依赖可靠 evaluator、外部 checker、测试、人工确认或至少可追溯的 trajectory。

写回：

- [x] 已理解错误反馈的风险；标准表达是“错误 feedback 会污染 experience，并被下一轮 Actor 复用”。

### Q4：为什么说 Reflexion 是“语言层的学习”，但不是模型权重训练？

我的回答：

并没有涉及到模型权重训练，这现在是一个用于agent loop的机制。

反馈：

这题答到了“不改权重”，但还要补出“为什么仍然叫学习”。

Reflexion 的学习发生在语言和上下文层：系统把失败反馈写成自然语言经验，再把这段经验放进下一轮 Actor 的上下文或记忆里。模型权重没有变化，没有梯度更新，也没有重新训练；变的是下一轮模型看到的信息和行动约束。

所以它像“拿了一张复盘便签再做一遍”，不是“把这次经验训练进模型参数”。这也是它轻量但脆弱的原因：便签写错，行为也会被带偏。

写回：

- [x] 已掌握不改权重；补强句：Reflexion 通过经验文本改变下一轮上下文，而不是通过训练改变模型参数。

### Q5：一条反思什么时候适合写入长期记忆，什么时候只适合留在当前任务上下文？

我的回答：

比如一个问题需要明确用到某条数学公式去解决某类问题，那么就适合写入长期记忆；比如一个问题中某个函数调用缺少了参数，某个思考步骤不对，那么就适合留在当前任务上下文。

反馈：

这个回答已经有很好的边界感：可泛化的规则更适合长期记忆，局部错误更适合留在当前任务上下文。

再补两条判断标准：

- 适合长期记忆：反馈可靠、已经被验证、能泛化到一类任务、不会泄露敏感信息、不和更高优先级规则冲突。
- 只适合当前上下文：一次性工具参数、当前 repo 的临时错误、当前网页状态、偶然失败、未验证猜测、敏感或过期信息。

小提醒：数学公式本身更像知识记忆；Reflexion 里的长期经验通常更像“下次遇到这类题，先识别适用条件，再选择公式”，也就是带有失败教训或策略改进的经验。

写回：

- [x] 已能区分长期经验和当前上下文；候选写回可保留到 [[Long-term Memory]] / [[Memory Reflection]]。
