# 09 概念层级审计基线

这页是概念层级归属审计的 vault 内稳定镜像。它保留给人和未来 agent 阅读的逐卡审计基线；机器可复跑的完整 JSON 保存在 `reports/concept-card-relation-map/`。

边界：这页不是新的事实来源，也不是写 `up` 的授权清单。真正写回仍必须走候选生成 / LLM 判定 / dry-run / limited apply。

项目规则已把本页作为新增或更新概念关系时的前置门禁：先对照本页判断 strict taxonomy、terminal no-parent 或 deferred-with-backlog，再运行 `scripts/concept_taxonomy/` 生成 dry-run / 验证报告。

## 以后新增概念卡怎么使用这条基线

1. 先做中英文术语审计，确认 canonical name，不把相邻概念写进 aliases。
2. 新卡写完后必须进入概念层级归属判断：是否属于已审计稳定父类、是否只是支撑/组成/代表/学习顺序/普通相关。
3. 若 strict taxonomy 成立，先生成候选，再 LLM 逐条判定，再 dry-run，最后 limited apply；不能直接写 `up`。
4. 若没有安全父类，必须写明 terminal reason 或进入 deferred-with-backlog；不能默默留空。
5. 若触碰本页的 24 个 deferred 边界，只能显式重开对应边界，不能把 suppressed target 当默认父类。
6. 新增卡后，机器基线中的 `total_concepts` 应随下一次审计同步增长；如果新卡没进入审计台账，就是未闭环。
