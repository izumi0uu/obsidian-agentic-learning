## 判断标准

- 它是新能力，还是新包装？
- 它依赖模型进步，还是工程系统改进？
- 它在什么任务上有效？
- 它失败时会怎样失败？
- 它是否改变我对 [[Agent]]、[[Memory]]、[[Tool Calling]] 或 [[Evaluation]] 的理解？

## 现代性边界

不是所有“现代系统怎么做”的问题都属于前沿。遇到旧论文范式、prompt pattern 或 agent loop 的现代化问题时，先按 [[LLM Wiki 工作流#操作 6：现代性 / 前沿性判定]] 分类：

- 经典论文和稳定概念放回 `wiki/concepts/`，作为基础地基。
- “过去靠 prompt 裸跑，现在被框架接管”的部分标成历史过渡，不追踪成前沿。
- 被多个框架共同采用的工程吸收方式，写回概念卡的现代系统段落。
- 只有具体 SDK/API、协议、产品能力、评测榜单或快速变化实现，才放进这里持续观察。

边界例子：[[ReAct]] 不是因为老就失效；裸 ReAct prompt loop 更像历史过渡；tool schema、state graph、guardrails、trace 和 eval 是当前工程实践；某个 Agents SDK 的具体 tracing API 才是易变前沿。

## 仍需观察

大部分第一批前沿候选已经做成概念卡。现在只保留真正还不稳定的词：

- [[Hermes Agent]]：2026-05-11 录入为 volatile 项目卡。观察重点不是“它是否重新定义 Agent”，而是 self-improving memory / skills、messaging gateway、MCP、security approval、durable multi-agent Kanban 和 `/goal` 在真实任务里的可靠性边界。
- [[OpenClaw Repo]]：2026-05-13 录入为 volatile source。观察重点不是 “Claw Bot” 这个别名，而是 local-first Gateway、多渠道入口、workspace memory、skills、sandbox、security audit、multi-agent routing 和 background tasks 如何组成 personal assistant harness。
- [[Agent Workflow Static Verification]]：2026-05-14 录入为 watch 概念卡。观察重点不是 Agentproof 这个工具是否立刻成为标准，而是 workflow graph lint / pre-deployment safety verification 是否会被 LangGraph、CrewAI、AutoGen、ADK 或 CI / observability 工具吸收。
- [[Workflow Guardrails]]：2026-05-14 录入为 current-practice / watch 概念卡。观察重点不是某个 guardrail SDK，而是 guardrail placement 是否继续向 workflow middleware、tool wrapper、validator、transaction、state hook 和 static check 收敛。
- [[LangChain DeepAgents]]：2026-05-11 录入为 volatile SDK / harness 卡。观察重点是 LangChain 是否继续把 long-horizon agent 的 planning、filesystem、subagents、memory、permissions 收敛成 Deep Agents SDK，还是回流为普通 LangGraph patterns。
- [[RAGGraph]]：目前更像项目名、RAG 工作流图或图知识库检索的混合说法，边界不如 [[GraphRAG]]、[[Agentic RAG]] 和 [[Agentic Retrieval]] 稳定。

后续遇到新词，不先建页；先在这里写一条观察，确认它是概念、协议、产品、项目名、工程能力还是风险模型。

## 2026-05-14 Workflow Guardrails

来源：

- [[Workflow Guardrails 主源]]
- [[Prefect Workflow Control Points]]
- [[Workflow Guardrails]]
- [[Workflow Guardrails 与 Prefect 控制点映射]]

一句话：

Workflow guardrails 把安全、质量、权限、格式和副作用控制放在 input、retrieval、model/tool call、validated output、transaction commit、state hook 和 automation 这些 workflow 边界上。

解决的旧问题：

它纠正“guardrails 等于最终回答过滤”的浅理解。对 Agentic workflow 来说，风险常发生在最终回答之前：检索片段进上下文、tool 参数提交、内部 API 调用、LLM 结构化结果写库、失败状态被吞掉。guardrail placement 要贴近这些边界。

可能的夸大：

这不是某个 SDK 已经统一解决的安全方案。OpenAI、LangChain、NeMo、Bedrock、Guardrails AI、Semantic Kernel、ADK、Prefect 都提供了不同控制点；真正实现仍要结合业务 policy、schema validator、transaction、approval、trace 和测试。

我现在不懂的点：

- tool guardrail 应该在 model tool call、internal API wrapper、DB write 三层都做，还是按风险分层？
- validation failure 应该优先表现为 retry / reask、FAILED state，还是人工复核任务？
- static workflow guardrail 会不会成为 framework compile step 或 CI lint 的标准能力？

## 2026-05-11 Hermes Agent

来源：

- [[Hermes Agent Repo]]

一句话：

Hermes Agent 是 Nous Research 的具体 Agent runtime 项目，把 memory、skills、messaging gateway、MCP、tool registry、security approval、delegation 和 `/goal` 式持续执行组合在一起。

解决的旧问题：

它把“LLM 会说话”推进到“Agent 可以跨入口、跨会话、带工具和带状态地运行”，适合作为 [[Agent Harness]] 的具体观察样本。

可能的夸大：

README 里的 self-improving 不应理解为模型权重自动变强；更像 memory、skills、session search 和 workflow 层的经验复用。真正可靠性仍要看权限、sandbox、trace、测试和人工验收。

我现在不懂的点：

- skills 自我改进如何避免把错误流程长期固化？
- 多平台 gateway 的授权和审计在团队/生产环境中如何落地？
- `/goal` 与 durable multi-agent Kanban 的任务验收边界如何设计？

## 2026-05-14 Agentproof / Agent Workflow Static Verification

来源：

- [[Agentproof - Static Verification of Agent Workflow Graphs]]
- [[Agent Workflow Static Verification]]

一句话：

Agentproof 把 LangGraph、CrewAI、AutoGen、Google ADK 这类 framework 暴露出的 workflow graph 抽取成统一 typed graph，并用结构检查、witness trace 和 temporal policy monitor 在部署前发现 topology-level safety defects。

解决的旧问题：

它补上 runtime guardrails 的一个盲区：如果错误路径没有被测试或真实运行触发，内容级 guardrail 可能看不到；但 graph 上的 dead end、不可达 exit、绕过 human gate 或缺少 tool declaration 可以先被静态检查标出来。

可能的夸大：

这不是完整 Agent 安全证明，也不验证 LLM 输出语义。论文 benchmark 是 18 个作者构造 workflow，适合看工具检测能力，不适合作为生产缺陷率统计。

我现在不懂的点：

- 条件边和动态 graph mutation 在真实 LangGraph 项目中应该怎么保守建模？
- human gate coverage 应该依赖命名、annotation，还是框架原生 interrupt / approval metadata？
- 这类能力未来会进入 framework compile step、CI lint、observability replay，还是独立安全工具？

## 2026-05-13 OpenClaw Repo

来源：

- [[OpenClaw Repo]]
- [[OpenClaw Repo vs Hermes Agent]]

一句话：

OpenClaw 是一个 local-first personal AI assistant / Agent gateway 项目，把多渠道消息入口、Gateway 控制面、embedded agent runtime、workspace memory、skills、nodes、sandbox、security audit 和 background tasks 组合成一个可长期运行的个人 Agent 系统。

解决的旧问题：

它把“模型可以回答消息”推进到“Agent 可以从多个真实消息渠道被触发、路由到不同 workspace/session/agent，并在权限、sandbox、memory 和 task ledger 约束下执行”，适合作为 [[Agent Harness]] 和 [[Agent Control Plane]] 的具体观察样本。

可能的夸大：

OpenClaw 官方安全模型是 personal assistant / single trusted operator boundary，不是 hostile multi-tenant 安全边界。`clawbot` 也只是 legacy alias，不应单独录成稳定概念。

我现在不懂的点：

- Gateway / node trust model 在多人共享渠道里如何避免权限被混用？
- Markdown memory、memory wiki 和 dreaming 在长期使用中如何控制噪音与错误固化？
- OpenClaw 的 background tasks、cron、heartbeat 和 agent sessions 在真实长任务里如何配合验收？

## 2026-05-11 LangChain DeepAgents

来源：

- [[LangChain Deep Agents 官方文档]]
- [[LangGraph 官方文档]]

一句话：

LangChain DeepAgents 是建立在 LangGraph 上的长任务 Agent harness，把 planning、filesystem、subagents、memory、permissions 等常见结构封装成 Deep Agents SDK。

解决的旧问题：

它把“普通 tool calling loop 能跑简单任务，但长任务缺计划、状态、文件、权限和恢复”的问题往 harness / runtime 层收敛。

可能的夸大：

DeepAgents 不是新模型，也不是 LangGraph 的替代品。它降低长任务 Agent 的脚手架成本，但不自动解决评测、安全、数据隔离、幂等和生产治理。

我现在不懂的点：

- 默认 filesystem / memory 如何治理临时结果、长期记忆和敏感数据边界？
- subagents 的任务拆分何时提升质量，何时只增加成本和 trace 噪音？
- 什么时候应该直接写 LangGraph graph，而不是使用 DeepAgents harness？
