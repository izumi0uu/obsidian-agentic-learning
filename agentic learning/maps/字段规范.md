# 字段规范

这页定义 vault 的基础字段。以后新增笔记优先遵守这里，而不是临时发明字段。

## 核心字段

```yaml
---
type:
topic:
status:
created:
updated:
source:
evidence:
aliases:     # concept only：中文名、缩写、常见英文变体
last_checked:
freshness:
conflicts:
related:
up:          # concept only：严格上位概念 / taxonomy
relations:   # concept only：非 taxonomy 的 typed relations
---
```

## type

`type` 说明这篇笔记是什么。

- `source`：外部资料、网页、论文、视频。
- `concept`：概念卡。
- `map`：导航页、主题地图、路线图。
- `experiment`：实验记录。
- `review`：概念触发式复习、费曼回答、学习校准记录。
- `frontier`：前沿追踪条目。
- `project-index`：项目或实验集合索引。
- `index`：vault 入口页。

## source_type

`source_type` 只用于 `type: source` 的 raw 笔记。

- `paper`：论文。
- `docs`：官方文档。
- `repo`：GitHub 项目、示例代码、SDK 仓库。
- `article`：文章、报告、实践指南。
- `web`：一般网页剪藏。
- `video`：视频、课程。


### paper 专属字段

`source_type: paper` 的 raw note 允许增加这些字段，用来把论文原文、PDF、抽取文本和发表信息分开：

```yaml
pdf:          # 原始 PDF URL，或 raw/papers/assets/ 下的相对路径
arxiv:        # arXiv abstract URL，可选
doi:          # DOI，可选
venue:        # 会议 / 期刊 / 报告来源，例如 ACL 2023、NeurIPS 2023
pages:        # 正式页码范围，可选
extracted:    # raw/papers/extracted/ 下的抽取文本相对路径，可选
```

边界：`pdf` / `extracted` 只是帮助定位原文证据，不替代 `evidence`。如果概念卡引用论文，优先锚到 source note 的小节；精读后再补 PDF 页码、section、figure 或 table。不要在未精读时伪造段落级证据。

## topic

`topic` 说明这篇笔记属于什么主题。优先用小写英文，方便 Dataview 查询。

常用值：

- `agent`
- `llm`
- `rag`
- `memory`
- `evaluation`
- `obsidian`
- `workflow`
- `frontier`
- `review`
- `feynman`

## status

`status` 说明处理状态。

- `inbox`：刚收集，还没处理。
- `seed`：已经有初步理解。
- `growing`：正在补例子、边界和链接。
- `mature`：相对稳定，可以复习或引用。
- `review`：需要重新检查或更新。
- `active`：导航页或长期活跃页面。

## source

`source` 只放来源链接或来源笔记。

概念卡里的 `source` 不等于参考文献列表。它回答：“这张卡最初从哪里长出来？”

## evidence

`evidence` 放更细的证据锚点。它回答：“这张卡里的关键理解，最少可以回到哪几个 source 小节或原文位置？”

优先格式：

```yaml
evidence:
  - "Source Note#为什么收"
  - "Source Note#需要我读的内容"
  - "PDF p.3 / Section 2.1"
```

边界：`source` 是来源列表，`evidence` 是最小证据锚点。没有精读到页码时，先用 source note 的小节锚点，不要假装有段落级引用。

## aliases

`aliases` 主要用于 `type: concept` 的概念卡，记录同一概念的中文名、缩写和常见英文变体。它服务中英文术语对齐和 Obsidian 搜索，不替代正文里的边界解释。

推荐格式：

```yaml
aliases:
  - 多路召回
  - 多路检索
  - multi-retriever retrieval
```

边界：

- `aliases` 只能写“同一概念”的别名；相邻概念、上位概念、组合策略、代表算法不要塞进 aliases。
- 英文 canonical name 不确定时，不要先建卡再用 aliases 圆回来；先写入 [[08 面试题概念卡待补充]] 或 [[05 Query 写回队列]]。
- 面试题自动链接还需要同步 `scripts/interview_question_concept_aliases.json`；frontmatter aliases 不会自动更新脚本。
- false friend 不得进入 aliases。例如 JVM memory 不是 Agent [[Memory]] 的 alias，Netty Reactor 不是 [[ReAct]] 的 alias。

## last_checked / freshness

`last_checked` 用于会更新的来源和前沿概念，格式用 `YYYY-MM-DD`。

`freshness` 用于判断下次是否要复查：

- `stable`：论文、基础概念、历史资料，不需要频繁查新。
- `watch`：前沿方向、框架、repo、协议，建议每月复查。
- `volatile`：API、产品文档、安全榜单、快速变化项目，建议每 1-2 周复查。
- `stale`：已经发现可能过期，优先复查。

## conflicts

`conflicts` 记录潜在矛盾或待核对说法。

```yaml
conflicts:
  - "OpenAI 和 Anthropic 对 Agent 定义边界不同，待比较。"
```

没有矛盾时可以留空数组或省略。发现矛盾后，不要直接覆盖旧说法；先写入 [[06 Wiki 健康检查]] 或 [[02 问题池]]。

## related

`related` 放最重要的 2-6 个相关双链。

不要把所有相关页面都塞进去。太多 related 会让地图变成噪音。

## 概念关系字段

概念卡可以额外使用 `up` 和 `relations` 来表达概念之间的结构。它们服务“理解关系”，不是为了把物理文件夹改成树。真实文件仍然平级放在 `wiki/concepts/`，逻辑层级由字段和链接表达。

### up

`up` 表示严格上位概念 / strict taxonomy / belongs-to。它回答：“这张概念卡属于哪个更大的概念类？”

推荐格式：

```yaml
up:
  - "[[Sparse Retrieval]]"
```

边界：

- 可以用于：`BM25 up [[Sparse Retrieval]]`，因为 BM25 是 sparse retrieval 的典型算法 / 方法族成员。
- 不用于学习顺序：不要因为“先学 TF-IDF 再学 BM25”就写 `BM25 up [[TF-IDF]]`。
- 不用于代表关系：代表关系写进 `relations: representative_of`，即使它和 `up` 指向同一个上位概念。
- 不用于思想来源：BM25 延续 TF-IDF 的词项权重直觉，但不是 TF-IDF 的下位概念。
- 不用于组合关系：Hybrid Search 组合 sparse 与 dense retrieval，不是 Sparse Retrieval 的子概念。
- 不用于普通相关：常一起出现但没有从属关系的概念放 `related` 或 `relations: related_to`。

插件边界：

- Abstract Folder：把 parent property 配成 `up` 后，可以把 `up` 渲染成虚拟文件夹树。
- Breadcrumbs：默认就有 `up` edge field，可以读取 YAML frontmatter 中的 `up`。
- Juggl：通过 Breadcrumbs integration 间接显示 `up/down` 关系；Juggl 不决定字段语义。

### relations

`relations` 表示非 taxonomy 的 typed relations。它回答：“这两个概念为什么有关？”

推荐格式：

```yaml
relations:
  - type: representative_of
    target: "[[Sparse Retrieval]]"
    note: "BM25 是 sparse retrieval 的常见代表算法。"
  - type: based_on_intuition
    target: "[[TF-IDF]]"
    note: "BM25 延续 TF-IDF 的词项权重直觉，但不是 TF-IDF 的下位概念。"
```

第一版关系词表：

- `representative_of`：代表 / 典型实现。例：BM25 是 Sparse Retrieval 的常见代表。
- `based_on_intuition`：基础直觉或思想来源。例：BM25 延续 TF-IDF 的词项权重直觉。
- `composes_with`：组合使用另一类方法。例：Hybrid Search 组合 Sparse Retrieval 与 Dense Retrieval。
- `composed_into`：被某个组合策略吸收。例：Sparse Retrieval 被组合进 Hybrid Search。
- `related_to`：相关但非从属，尤其用于容易混淆或常一起出现的概念。

`relations` 与 `related` 的区别：

- `related` 是少量普通邻接链接，适合 2-6 个重要相关页面。
- `relations` 用于高混淆关系，必须写 `type + target + note`，说明关系类型和原因。
- 不要把所有双链都搬进 `relations`；只有“关系类型本身能帮助理解边界”时才写。

插件边界：`relations` 的嵌套 YAML 对人类、LLM 和 Dataview 有用，但不是 Breadcrumbs 的顶层 edge field 结构，Juggl 也不会天然把它画成边。若未来确实需要在图上可视化 `based_on` / `composes_with` 等关系，再单独增加少量顶层镜像字段；第一版不启用，避免字段爆炸。

### children

`children` 不作为普通概念卡的常规字段。反向子概念优先由子卡的 `up` 推导出来：

- Breadcrumbs 可以从 `up` 推导 implied `down`。
- Abstract Folder 可以通过 `up` 渲染父子树。

只有在非 Markdown 附件、Canvas、Excalidraw、PDF，或确实需要父卡显式收纳时，才考虑手写 `children`。不要为了让父卡“看起来完整”而手动维护 children 列表，否则容易和子卡 `up` 漂移。

### 示例：TF-IDF / BM25 / Sparse Retrieval / Hybrid Search

`BM25`：

```yaml
up:
  - "[[Sparse Retrieval]]"
relations:
  - type: representative_of
    target: "[[Sparse Retrieval]]"
    note: "BM25 是 sparse retrieval 的常见代表算法；这里的代表关系比 up 更具体。"
  - type: based_on_intuition
    target: "[[TF-IDF]]"
    note: "BM25 延续 TF-IDF 的词项权重直觉，但加入长度归一化和饱和控制。"
  - type: related_to
    target: "[[Hybrid Search]]"
    note: "Hybrid Search 常把 BM25 / sparse retrieval 作为一路召回。"
related:
  - "[[TF-IDF]]"
  - "[[Sparse Retrieval]]"
  - "[[Hybrid Search]]"
```

`Hybrid Search`：

```yaml
up:
  - "[[Retrieval]]" # 仅当 vault 中已有稳定的 Retrieval 上位概念时使用；否则先留空
relations:
  - type: composes_with
    target: "[[Sparse Retrieval]]"
    note: "Hybrid Search 通常把 sparse retrieval 作为一路召回或排序信号。"
  - type: composes_with
    target: "[[Dense Retrieval]]"
    note: "Hybrid Search 通常把 dense retrieval / embedding search 作为另一条语义召回。"
related:
  - "[[BM25]]"
  - "[[Sparse Retrieval]]"
```

如果 `[[Retrieval]]` 或 `[[Dense Retrieval]]` 尚未成为稳定概念卡，不要为了字段完整而伪造强链接；可先写在正文、[[02 问题池]] 或 [[05 Query 写回队列]]。

## 边界规则

- 目录决定文件物理位置。
- `type` 决定这篇笔记的角色。
- `topic` 决定它属于哪条知识线。
- `status` 决定下一步要不要整理。
- `related` 决定它在图谱里怎么连接。
- `evidence` 决定这张卡是否能追溯到来源。
- `freshness` 决定它是否需要定期复查。
