# 插件配置

当前策略：先用 Obsidian 核心能力 + Web Clipper + Dataview + Templater；概念层级显示主要用 Abstract Folder / Breadcrumbs。Juggl 因性能问题已退出后续方案，不再作为临时启用或开发目标。AI 问答等笔记量上来再接。

Codex 侧已经有一个本地 skill：

```text
~/.codex/skills/obsidian-llm-wiki
```

它会读取本项目的 `AGENTS.md`、[[LLM Wiki 工作流]] 和 [[字段规范]]，再执行 ingest/query/lint。

## 必装

### Obsidian Web Clipper

用途：把网页和元数据保存为 Markdown。

建议默认保存到：

```text
raw/inbox/
```

建议使用模板：

```text
templates/网页剪藏.md
```

安装边界：Web Clipper 是浏览器扩展，不是 vault 内的社区插件。vault 端已经准备好 `raw/inbox/` 和 `templates/网页剪藏.md`，浏览器扩展本身需要在浏览器扩展商店或 Obsidian 官方页面安装。

### Dataview

用途：把 frontmatter 字段变成可查询索引。

安装后，`index.md` 和 [[Agent 知识地图]] 里的 dataview 代码块会自动显示表格。

当前状态：插件文件已安装；如果 Obsidian 里表格没有显示，到设置里确认 Community plugins 是否启用了 Dataview。

### Templater

用途：在模板里自动填日期、标题和元数据。

当前状态：插件文件已安装；如果新建模板没有自动日期，到设置里确认 Community plugins 是否启用了 Templater。模板目录为：

```text
templates/
```

当前策略：先不开 folder template 和 system command。等模板稳定后再增加自动化，避免新手期被自动规则干扰。

## 概念层级显示插件

当前 vault 的原则是：物理文件夹保持扁平，概念层级写在 frontmatter 和链接关系里。插件只负责显示，不负责决定知识语义。

### Abstract Folder

用途：把概念卡的 `up` 字段渲染成虚拟文件夹树，让 `wiki/concepts/` 里的平级文件可以按逻辑上位概念浏览。

当前状态：已本地安装并启用，manifest 版本为 `1.14.0`。本 vault 的配置文件在：

```text
.obsidian/plugins/abstract-folder/data.json
```

当前写入的关键配置：

```json
{
  "propertyName": "up",
  "parentPropertyNames": ["up"],
  "childrenPropertyName": "children",
  "childrenPropertyNames": ["children"],
  "showAliases": false,
  "displayNameOrder": ["title", "basename"],
  "excludedPaths": [
    "raw/",
    "maps/",
    "reviews/",
    "templates/",
    "Excalidraw/",
    "assets/",
    "wiki/topics/",
    "wiki/projects/",
    ".obsidian/",
    ".trash/",
    "index.md",
    "log.md"
  ]
}
```

排除策略：第一版 Abstract Folder 只服务 `wiki/concepts/` 的概念 taxonomy 视图；`excludedPaths` 按路径前缀过滤，所以 `raw/` 会把面试题、论文摘录、repo source note 一起排除，`maps/` 会把工作流地图、索引和健康检查页排除。这不会删除或移动文件，只是不让它们进入 Abstract Folder 的虚拟树 / 图索引。

保留边界：如果以后希望 topic 页也进入概念树，可以从 `excludedPaths` 中移除 `wiki/topics/`；当前先排除它，是为了避免主题页、项目页、入口页和日志混进概念卡层级。

显示名策略：Abstract Folder 按 `displayNameOrder` 从左到右找显示名；当前不再把 `aliases` 当默认显示名来源，而是优先 `title`，再退回到文件名 `basename`。这样英文概念卡会直接显示英文文件名，中文别名继续留在 frontmatter 里供搜索和语义对齐使用。少数旧卡如果需要英文显示名，可以补 `title` 作为显式覆盖。

生效注意：如果你在 Obsidian 运行时直接改 `.obsidian/plugins/abstract-folder/data.json`，插件内存里的旧设置可能在关闭或保存时回写覆盖。最稳的做法是通过插件设置面板修改；如果必须外部改文件，先退出 Obsidian，再改完重新打开 vault。

建议设置：

```text
Property Name = up
Children Property Name = children
Display Name Priority = title, basename
Show Aliases = off
```

边界：

- 不新增重复的 `parent:` 字段；本 vault 用 `up` 作为唯一第一版 taxonomy 字段。
- 不运行 “Convert folder structure to plugin format”，因为当前物理目录不是概念 taxonomy。
- 第一阶段把 Abstract Folder 当阅读 / 浏览视图，不默认用拖拽批量改层级；拖拽会直接改 frontmatter，容易把临时浏览动作变成知识结构事实。
- 普通概念卡不手写 `children`，反向子概念由子卡 `up` 和插件视图推导。

### Breadcrumbs

用途：读取 typed links / edge fields，为概念卡提供上位、下位和路径导航。

当前状态：已本地安装并启用，manifest 版本为 `4.9.5`。第一版依赖 Breadcrumbs 默认 edge fields，不额外新增 `based_on` / `composes_with` 等镜像字段。

建议设置：

```text
保留默认 edge fields: up / down / same / next / prev
确认 up 属于 ups field group
修改概念卡 frontmatter 后运行 Breadcrumbs: Rebuild Graph
```

边界：

- 子卡写 `up` 即可；父卡不需要手写 `down`。Breadcrumbs 可以通过 implied edges 推导反向关系。
- `relations:` 是嵌套 YAML 解释层，不是 Breadcrumbs 顶层 edge field；第一版不要期待 Breadcrumbs 直接渲染 `relations`。
- 暂不把普通 `related:` 注册为 Breadcrumbs edge field，避免图谱噪音。

### Juggl（已退役）

历史用途：把 Obsidian / Breadcrumbs 关系可视化成可交互局部图。

当前状态：因卡片多时明显卡顿，已从后续工作流和开发计划中退役；不要重新启用、安装或围绕 Juggl 设计字段。若某台机器仍残留 `.obsidian/plugins/juggl/` 目录或旧 manifest，只视为历史残留，不代表当前方案仍使用 Juggl。

后续边界：

- 不再通过 Juggl 查看 `up/down` taxonomy 图；主路径是 Abstract Folder 虚拟树 + Breadcrumbs typed edges。
- 不为了 Juggl 新增 `based_on` / `composes_with` / `related_to` 等顶层镜像字段。
- 不依赖 Breadcrumbs v4 已移除的旧 Juggl view / codeblock，也不从 Juggl 插件侧恢复图视图。

边界：第一版概念层级只依赖 Abstract Folder / Breadcrumbs。`based_on_intuition`、`composes_with` 等关系先保留在 `relations:` 里；只有当这些关系稳定且确实需要在 Breadcrumbs 中图形化时，再考虑增加顶层镜像字段。
