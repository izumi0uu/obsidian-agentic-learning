# 常用向量数据库对比

## 一句话总览

常用向量数据库 / 向量检索选择不是一条 vendor 排名，而是一组工程层选择：本地库、Postgres 扩展、专用向量服务、搜索系统向量检索、图数据库 + 向量 / 全文索引分别适合不同阶段。

最小边界：[[Vector Database]] 负责存储和检索 embedding；本页记录常见实现生态和选型刀口，不替代具体版本 benchmark，也不把产品能力写成永久真理。

## 为什么这组值得单独录入

- 学习价值：Agent / RAG 入门时经常听到 Chroma、FAISS、pgvector、Qdrant、Milvus、Weaviate、Pinecone、Elasticsearch、OpenSearch、Neo4j，但它们并不在同一层。
- 混淆风险：FAISS 更像向量检索库，pgvector 是 PostgreSQL 扩展，Qdrant / Milvus / Weaviate / Pinecone 更像专用向量数据库 / 服务，Elasticsearch / OpenSearch 是搜索系统，Neo4j 是图数据库生态。
- 工程价值：选型会影响 metadata filter、权限、删除、更新、hybrid search、运维成本、数据同步、trace 和 RAG evaluation。
- 边界价值：把它们录成一页对比，比给每个产品建弱概念卡更稳；后续只有当某个产品进入项目实战或官方文档精读时，再建单独 source / concept 页。

## 为什么这组值得对比

这组值得对比，是因为“向量数据库”在工程里可能指本地向量库、Postgres 扩展、专用向量服务、搜索引擎向量能力、图数据库向量索引或托管 RAG 数据层。它们共同服务 [[Dense Retrieval]] / [[Vector Database]]，但在部署、过滤、权限、metadata、hybrid search、[[Vector Search Algorithm|向量搜索算法]]、运维和成本上差别很大。

## 共同问题域

共同问题域是：embedding 生成后，系统如何存储向量、保存 metadata、执行 top-k / [[Vector Search Algorithm]] / [[Approximate Nearest Neighbor Search|ANN]] / filter / hybrid search，并把候选交给 [[Reranking]]、[[Context Engineering]] 或 RAG answer 阶段。

```text
documents -> chunks -> embeddings -> vector index / database -> filtered top-k -> rerank/context -> answer
```

## 分类总览

| 类别 | 常见选择 | 最适合先问的问题 | 典型位置 |
|---|---|---|---|
| 本地 demo / 学习库 | Chroma、FAISS | 我能不能最快跑通 embedding -> search -> answer？ | notebook、本地原型、课程 demo、小型实验 |
| Postgres 扩展 | PostgreSQL + pgvector | 我已有业务库，能否把结构化数据和向量检索放一起？ | 小中型产品 MVP、已有 PostgreSQL 后端 |
| 专用向量数据库 / 服务 | Qdrant、Milvus、Weaviate、Pinecone | 我是否需要专门的向量索引、过滤、扩展、部署或托管能力？ | 生产 RAG、服务化检索、多租户或更大规模召回 |
| 搜索系统向量检索 | Elasticsearch、OpenSearch | 我是否已经有全文检索 / keyword search 栈，并希望做 hybrid search？ | 企业搜索、日志/文档搜索、关键词 + 向量融合 |
| 图数据库 + 向量 / 全文索引 | Neo4j | 我是否需要实体、关系、路径遍历和多跳关系？ | GraphRAG、知识图谱、关系密集型问答 |

## 核心区别表

| 名称 | 更准确的层级 | 官方 / 项目定位可支持的稳定判断 | 适合优先使用 | 不适合优先使用 |
|---|---|---|---|---|
| Chroma | AI-native 向量数据库 / 本地与云端数据层 | Chroma docs 以 document storage、vector search、Chroma Cloud 等能力组织文档 | 学习、快速 RAG demo、本地 collection 和轻量应用 | 还没定义权限、审计、SLA、复杂检索治理时，不要直接把 demo 配置当生产架构 |
| FAISS | 向量相似搜索 / 聚类库 | FAISS repo 明确是 dense vector similarity search 和 clustering library | 本地高性能向量索引、[[Vector Similarity Metrics|向量相似度]] / ANN 算法学习、实验检索、被其他系统作为底层索引 | 它不是完整数据库；metadata、权限、多租户、备份、服务 API 需要自己补 |
| pgvector | PostgreSQL 向量相似搜索扩展 | pgvector repo 定位为 Postgres 的开源向量相似搜索，保留 Postgres 的 ACID、JOIN、恢复和客户端生态 | 已有 PostgreSQL、MVP、结构化字段 + 向量检索、权限/业务数据同库 | 向量规模、QPS、复杂 ANN 调优或独立检索团队需求很高时，需要评估专用服务 |
| Qdrant | 专用向量数据库 / client-server 服务 | Qdrant docs 以 collection、point、payload、top-k similarity、部署方式组织 | 需要开源/可托管、payload filter、服务化向量检索、明确的 API 边界 | 不要只为小 demo 引入一整套服务；hybrid / ranking / eval 仍要单独设计 |
| Milvus | 开源、云原生向量数据库 | Milvus docs 以构建 vector similarity search applications 和大规模向量数据组织文档 | 大规模向量检索、开源生态、需要集群/服务化能力的场景 | 运维复杂度比本地库和 pgvector 高；小项目可能过重 |
| Weaviate | AI-native / open-source vector database | Weaviate docs 明确支持 vector、keyword、hybrid search、RAG、reranking、filters 等查询能力 | 希望检索形态集成度高，尤其需要 vector + keyword / hybrid / rerank / RAG workflow 的场景 | 如果团队只需要极简向量近邻搜索，集成能力可能反而增加概念负担 |
| Pinecone | 托管生产向量数据库 | Pinecone docs 定位为生产级 AI 应用的向量数据库，并强调 managed database、dense/sparse/full-text schema 等 | 不想自运维、需要托管服务、生产语义搜索、快速接入云服务 | 成本、数据驻留、vendor lock-in、离线/内网部署要求强时要谨慎 |
| Elasticsearch | 搜索引擎 + vector search | Elastic docs 有 vector search、dense vector、semantic_text 等搜索能力 | 已有 Elastic 搜索栈，需要全文、过滤、聚合和向量检索一起做 | 如果只做小规模语义检索，引入完整搜索系统可能太重 |
| OpenSearch | 搜索系统 + vector search / k-NN | OpenSearch docs 把 vector search 集成到索引、k-NN field、ANN / exact search 中 | 已有 OpenSearch，或需要开源搜索栈里的关键词 + 向量组合 | 同样不是“纯向量库替代品”；索引设置、版本和集群运维会影响复杂度 |
| Neo4j | 图数据库 + vector / full-text semantic indexes | Neo4j docs 支持 vector indexes；GraphRAG docs 提供图 + LLM 生态入口 | 实体关系、多跳路径、关系密集型 GraphRAG，需要 Cypher / graph traversal | 不要把 Neo4j 当“更高级的向量库”；构图、实体消歧、关系维护才是核心成本 |

比较锚点：[[Vector Database]]、[[Hybrid Search]]、[[Knowledge Graph]]、[[Neo4j]]。
## 选型路径

```text
只是学习 / 本地 demo?
  -> 先 Chroma 或 FAISS

已有 PostgreSQL 后端，数据量和 QPS 中等?
  -> 先评估 pgvector

需要独立检索服务、扩展、过滤、多租户或托管?
  -> 比较 Qdrant / Milvus / Weaviate / Pinecone

企业已有 Elasticsearch / OpenSearch?
  -> 先评估同栈 vector search + keyword/full-text + metadata filter

问题以实体关系、多跳路径、图遍历为主?
  -> 评估 Neo4j + vector/full-text index + GraphRAG
```

小边界：这个路径不是性能结论，而是“先排除明显不匹配的层”。真正选型还要用自己的语料、query、权限模型、更新频率和评测集跑实验。

## Agent / RAG 场景下的判断刀口

| 判断维度 | 要问的问题 | 为什么影响选型 |
|---|---|---|
| 数据规模 | 是几千、几百万还是上亿 vectors？ | 小规模可以本地或 pgvector；大规模服务化可能需要专用向量数据库 |
| 更新频率 | 文档和 memory 是否频繁增删改？ | 删除、版本、过期、重建索引和一致性会影响生产可用性 |
| metadata / 权限 | 是否要按 tenant、用户、文档权限、时间、状态过滤？ | Agent / RAG 很容易因为检索前过滤缺失而越权 |
| hybrid search | 是否有专有名词、编号、错误码、代码符号？ | 纯向量容易漏精确匹配；搜索栈或支持 hybrid 的系统更合适 |
| 运维边界 | 团队愿意自运维还是更偏托管？ | Pinecone 等托管服务减少运维；Milvus / Qdrant / Weaviate / OpenSearch 等需要看部署形态 |
| 数据边界 | 数据能否出内网 / 出云 / 进第三方服务？ | 影响托管服务、私有部署和合规选择 |
| 现有栈 | 已经有 Postgres、Elastic/OpenSearch、Neo4j 吗？ | 复用现有系统可能比引入新数据库更便宜、更易治理 |
| evaluation | 有没有 retrieval eval / citation eval？ | 没有评测时，换向量库常常只是换一种不确定性 |

## 常见误解

- 误解：Qdrant、Milvus、Weaviate、Pinecone、pgvector、FAISS、Chroma 是同一层产品，只需要选“最强的”。
  - 边界：它们跨越 library、database extension、专用服务、托管服务和搜索 / 图数据库生态。
- 误解：用了向量数据库就有可靠 RAG。
  - 边界：可靠 RAG 还需要 ingestion、chunking、metadata、权限、hybrid search、reranking、citation 和 evaluation。
- 误解：Agent 长期记忆都应该存在向量库。
  - 边界：用户偏好、配置、权限、状态、审批、删除策略更适合关系库、KV、事件日志或 state store；向量库适合语义召回的摘要、文档 chunk、经验片段。
- 误解：GraphRAG 就是 Neo4j，Neo4j 就是向量数据库。
  - 边界：GraphRAG 是图结构参与检索和上下文构造的模式；Neo4j 是图数据库生态，可以结合 vector / full-text index。

## 现代性状态

- 判定：current-practice / watch。
- 稳定部分：Agent / RAG 系统需要把向量检索视为 retrieval infrastructure，而不是模型能力本身。
- 易变部分：产品能力、API、索引实现、hybrid search 支持、托管方式、价格、benchmark 和版本兼容。
- 复查节奏：涉及具体产品能力时，至少在项目选型前查一次官方文档；本页 `last_checked: 2026-05-15` 只表示本页证据复查日期。

## 不单独建概念卡的边界

当前不为 Chroma、FAISS、pgvector、Qdrant、Milvus、Weaviate、Pinecone、Elasticsearch、OpenSearch 分别创建稳定概念卡，原因是：

1. 目前学习目标是区分“常用向量数据库生态层级”，不是精读某一个产品。
2. 产品能力变化快，单独概念卡容易过期。
3. 真正需要单页时，应以官方 docs source note、项目实战记录或 benchmark 结果为证据，而不是只凭名称热度。

例外：[[Neo4j]] 已经作为 GraphRAG / Knowledge Graph 工程生态有单独概念卡，因为它在本 vault 中还承担图数据库与 GraphRAG 边界的学习价值。

## 最容易混淆的边界

- [[Vector Database]] 不等于 [[Embedding]] 模型；前者存和查向量，后者生成向量。
- FAISS 这类库不等于完整数据库；metadata、权限、备份和服务 API 要自己补。
- 图数据库的向量索引不等于完整 [[GraphRAG]] 方法；它只是图/向量组合的一种基础设施。
- 支持 hybrid search 的产品不自动解决 reranking、citation faithfulness 或 access control。

## 执行时序 / 机制差异

向量数据库通常处在 ingestion 后、rerank/context 前：先写入 chunk、embedding 和 metadata；查询时用 query embedding 按某种 [[Vector Similarity Metrics]] 检索候选；再用 payload/metadata filter、全文/稀疏路线、rerank 或业务规则缩小上下文。不同产品的差异主要在 [[Vector Search Algorithm|索引 / 搜索算法]]（例如 [[Approximate Nearest Neighbor Search|ANN]] / [[HNSW]]）、过滤能力、服务化、事务/备份、多租户和与搜索/图/SQL 生态的组合方式。

## 学习类比（非证据）

可以把这些实现类比成不同类型的仓库：FAISS 像高性能货架算法，pgvector 像把向量货架放进已有 Postgres 仓库，Qdrant/Milvus/Pinecone 像专门的向量仓储服务，Elastic/OpenSearch 像搜索仓库加向量货架，Neo4j 像关系地图旁边挂向量索引。

类比边界：这只是学习类比（非证据），不代表官方产品定位，也不替代版本 benchmark、SLA 或部署文档。

## 现代系统如何吸收或限制

来源支持的是各产品/项目提供向量检索、过滤、索引或搜索组合能力；Embeddings lesson 补充了相似度度量、ANN / HNSW、向量压缩和 benchmark 的机制边界。工程综合 / inference 是：现代 RAG 往往不会只靠一个 vector DB 完成质量闭环，而会把 embedding、chunking、metadata、hybrid retrieval、rerank、evaluation、access control 和 observability 接在一起。限制在于产品能力和托管形态变化快，本页选型判断需要按 official docs 复查。

## 什么时候用哪个判断

- 学习/本地实验：优先 Chroma、FAISS、sqlite/本地方案或轻量服务。
- 已有 Postgres 和结构化业务数据：优先评估 pgvector。
- 需要独立向量服务、payload filter、扩展和 API 边界：评估 Qdrant、Milvus、Pinecone、Weaviate 等。
- 已有搜索栈并重视全文 + 向量组合：评估 Elastic / OpenSearch。
- 知识关系和多跳图查询是核心：评估 Neo4j + vector/fulltext，而不是只看向量能力。

## 它们共同不是什么

它们共同不是 RAG 的全部，也不是答案质量保证。向量数据库不能替代 embedding 选择、chunking、query rewrite、reranking、citation faithfulness、权限过滤或人工评估；它只解决“向量候选如何存和查”的基础设施问题。
