# BM25

## 一句话

BM25 是一种经典关键词检索 / [[Sparse Retrieval]] 打分函数，常用于搜索引擎和 RAG 的 sparse side，擅长命中产品型号、专有名词、错误码、函数名、编号这类精确词面信号。

## 概念详解

BM25 的全名常写作 Okapi BM25。学习时可以把它理解成 [[TF-IDF]] 直觉的工程化版本：它仍然看一个词在文档里是否常出现，也看这个词在整个语料里是否稀缺；但它不会让某个词因为重复出现很多次就无限加分，还会考虑文档长度等因素，让排序更适合真实搜索。

在 RAG 里，BM25 的位置通常不是生成答案，也不是理解语义，而是在线检索阶段的一路候选召回。用户 query 先被分词或 token 化，搜索系统在倒排索引里找到包含查询词的文档或 chunk，再按 BM25 分数排序。它对 `M4 Pro`、`LSTM`、`ERR_CONNECTION_RESET`、`invoice_id` 这种必须精确出现的词很敏感。

BM25 的盲区也很清楚：它不理解“退货”和“申请售后”这类同义或近义表达。如果 query 和文档没有词面重叠，BM25 可能给不到分。因此现代 RAG 通常把 BM25 和 dense [[Embedding]] 检索并行：BM25 保精确词，向量检索保语义相似，再用 RRF、去重和 [[Reranking]] 合并排序，形成 [[Hybrid Search]]。

证据边界：xiaolinnote 的关键词检索 source note 支持 BM25 作为 Elasticsearch、Lucene 等传统搜索引擎核心算法的代表；多路召回 source note 支持 BM25 作为 Sparse Retrieval 的第二路；TF-IDF 概念卡支持 BM25 与 TF-IDF 的边界。

## 它解决什么问题

BM25 解决的是“包含关键查询词的文档应该排在前面，但不能被常见词和无限重复词误导”的问题。

没有 BM25 这类 sparse ranking 时，纯向量检索可能找来语义接近但没有精确实体的 chunk；只用原始词频又会被常见词和重复词拖偏。BM25 把词频、词项稀缺度、词频饱和和文档长度等因素合在一起，让关键词检索更稳。

## 它不是什么

BM25 不是 [[TF-IDF]] 的同义词。它继承了 TF / IDF 的直觉，但增加了更适合搜索排序的修正。

BM25 不是整个 [[Sparse Retrieval]]。它是 sparse retrieval 的常见代表，不等于全文检索、倒排索引或所有稀疏神经检索方法。

BM25 也不是 [[Embedding]]。它不把文本映射到 dense semantic space，不负责理解同义词、隐含关系或跨语言语义。

## 最小例子

```python
from rank_bm25 import BM25Okapi

corpus = [
    ["苹果", "手机", "截图", "方法"],
    ["iPhone", "截屏", "教程"],
    ["安卓", "手机", "拍照"],
]

bm25 = BM25Okapi(corpus)
query = ["苹果", "手机", "截图"]
scores = bm25.get_scores(query)
```

这个例子里的核心不是 Python 库本身，而是 BM25 的检索形态：文档和 query 都先变成词项列表，再按词面重叠与词项权重打分。

RAG 里更常见的落点：

```text
query
  -> BM25 keyword retrieval top 50
  -> vector retrieval top 50
  -> RRF merge
  -> rerank
  -> answer with citations
```

## 常见误解 / 风险

- 以为 BM25 已经过时：它不负责语义理解，但在精确词面检索上仍然是强基线。
- 以为 BM25 比向量检索更“正确”：BM25 遇到同义词、改写和语义相关但词不重叠的文档会失效。
- 忽略分词：中文、代码符号、大小写、连字符和缩写会直接影响 BM25 召回。
- 直接把 BM25 分数和向量相似度相加：二者分数量纲不同，通常需要 RRF、归一化或 rerank。
- 只看 BM25 top-k：精确词命中不等于 chunk 足以回答问题，还要看引用支持和上下文完整性。

## 边界细节

和 [[TF-IDF]] 的边界：TF-IDF 是基础词项权重表示；BM25 是更常用于搜索排序的相关性函数。

和 [[Sparse Retrieval]] 的边界：BM25 是常见代表；sparse retrieval 是家族。

和 [[Hybrid Search]] 的边界：BM25 常作为 hybrid search 的关键词/稀疏检索一路；Hybrid Search 还包括 dense retrieval、合并、去重、过滤和可能的 reranking。

和 [[Embedding]] 的边界：embedding 处理语义相似，BM25 处理词面匹配。二者互补，不是单向替代。

和 [[Reranking]] 的边界：BM25 常用于初召回或初排；reranker 在候选集合上做二阶段精排。

## 现代性状态

- 判定：foundation / current-practice。
- 稳定部分：BM25 作为关键词检索强基线和 sparse retrieval 代表非常稳定。
- 现代吸收方式：生产 RAG 常把 BM25 与 dense retrieval 并行，用 RRF 或 reranking 融合。
- 易变部分：具体搜索引擎实现、tokenizer、参数默认值、hybrid search API 和稀疏神经检索替代方案会变化。

## 现代系统怎么吸收 BM25 的价值

现代 RAG 系统吸收 BM25 的方式通常不是“回到传统搜索”，而是把它作为检索质量的兜底信号：当 query 里有专有名词、编号、代码符号、错误码或字段名时，BM25 候选可以防止 dense retrieval 漏掉字面证据。

一个可调试的系统应该能看到 BM25 top-k、vector top-k、融合后的候选、被 filter 去掉的候选、rerank 前后排序和最终进入上下文的 chunk。这样才能判断失败是 BM25 分词问题、dense retrieval 语义问题、融合问题，还是后续 rerank / generation 问题。
