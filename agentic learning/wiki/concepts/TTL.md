# TTL

## 一句话

TTL（Time To Live）是给缓存、记忆、索引记录、锁、心跳或网络数据设置“最多可被无条件相信多久”的生命周期边界；到期后，系统应删除、降权、刷新、重建或重新验证。

## 概念详解

TTL 的核心不是“定时删除”本身，而是把信息的有效性从无限期改成有边界。一个值被写入缓存、记忆库或检索索引时，如果没有任何过期策略，它可能在业务事实已经变化后继续被使用，变成更隐蔽的错误来源。TTL 给系统一个默认判断：超过某个时间点后，这条信息不能再被当作新鲜事实直接使用。

在缓存系统里，TTL 常表现为 key 的过期时间。比如空值缓存只保留 30 到 120 秒，避免不存在的 ID 被反复打到数据库；热点缓存如果全部设置同一个 TTL，可能在同一秒集中失效，引发缓存雪崩，所以常配合随机抖动。

在 Agent memory 里，TTL 更像“记忆治理”参数。长期记忆可以带来源、时间、版本、最后命中时间、重要度和 TTL；长期不命中或超过有效期的记忆可以自动降权、归档、清理或要求重新确认。这里 TTL 不是说记忆一定要机械删除，而是提醒系统旧偏好、旧项目事实和旧检索结果可能会污染当前任务。

在 RAG 里，TTL 可以用于知识库 freshness：文档、chunk、embedding、检索缓存和答案缓存都可能过期。政策、费率、价格、API 文档、网页内容这类高变化资料，应该有失效时间、版本或更新时间；否则 RAG 会把“外部知识可更新”的优势变成“旧索引继续误导模型”的风险。

工程上还要区分硬 TTL 和软 TTL。硬 TTL 到期后直接不可用，适合锁、幂等状态、空值缓存这类“留太久会伤害正确性”的状态；软 TTL 到期后先标记为 stale，再后台刷新或要求重新验证，适合 RAG 文档、长期记忆和语义缓存这类“旧信息仍可能有参考价值，但不能继续排在最高可信位”的内容。这个区别很小，却能防止一个常见误判：把 TTL 当成简单删除按钮。真实系统往往需要同时回答“还能不能读”“读了是否降权”“是否触发刷新”“是否需要向用户暴露不确定性”四个问题。

## 它解决什么问题

TTL 解决的是“信息什么时候不再默认可信”的问题。

没有 TTL，系统会倾向于保留一切：缓存永不失效，记忆越积越多，知识库旧版本继续被检索，分布式锁异常后可能长期占住资源。TTL 让系统能在无需人工逐条检查的情况下，把陈旧、低命中或高风险的信息排出关键路径。

## 它不是什么

TTL 不是强一致性机制。它只能限制旧值最多保留多久，不能保证到期前的数据一定正确，也不能保证刚更新的数据立刻覆盖所有缓存。

TTL 不是 LRU。TTL 按时间过期；LRU 按最近使用情况淘汰。一个频繁访问但超过 TTL 的值仍应被视为过期；一个没过 TTL 但很久没访问的值可能被 LRU 淘汰。

TTL 也不是事实校验。RAG 或 memory 里设置了 TTL，只说明资料到期后要重新检查；它不能判断资料本身是否真实、引用是否支持答案、权限是否正确。

网络包里的 TTL 也要单独看边界。它历史上叫 Time To Live，但在现代 IP 网络里通常表现为 hop limit：每过一个路由器递减，归零后丢弃，主要防止包在网络里无限循环。它和缓存有效期共享“有限生命周期”的思想，但不是同一个工程场景。

## 最小例子

```text
cache.set("user:123", user_profile, ttl=30min)
```

这表示 `user:123` 这份缓存最多默认可信 30 分钟。30 分钟后再访问，系统应重新读取数据库、刷新缓存，或返回旧值但明确进入降级策略。

RAG 场景可以写成：

```text
policy_chunk.updated_at = 2026-05-01
policy_chunk.ttl = 30d
```

如果今天已经超过有效期，retriever 可以降低它的排序、触发重新抓取，或在回答前要求引用更新版本。

## 常见误解 / 风险

- 误解：设置 TTL 就不会读到旧数据。实际上 TTL 到期前仍可能读到旧值；需要强一致时还要配合主动失效、版本号、消息通知或读写路径约束。
- 误解：TTL 越短越安全。太短会造成缓存频繁回源、索引反复重建、成本和延迟上升。
- 误解：TTL 越长越省事。太长会让旧知识、旧偏好或旧锁状态长期污染系统。
- 风险：大量 key 使用同一个 TTL，可能在同一时刻集中失效，引发缓存雪崩。
- 风险：RAG 只更新原文不更新 embedding / chunk metadata，会让索引层继续召回过期内容。

## 边界细节

TTL 通常有三种使用方式：

1. 直接删除：到期后 key、缓存结果或临时状态不可再读。
2. 降权保留：到期后不立刻删除，但在检索、排序或 memory 注入时降低优先级。
3. 触发刷新：到期后触发重新抓取、重新 embedding、重新确认或后台异步更新。

缓存系统更常用第一种和第三种；Agent memory 和 RAG 知识库常需要第二种和第三种，因为旧信息有时仍有参考价值，但不能继续占据最高可信位置。

概念层级边界：本卡不直接写 `up`。TTL 是横跨 cache、memory、RAG、lock、heartbeat 和 network 的生命周期参数，不是 [[Memory]]、[[RAG]] 或 [[Document Ingestion]] 的严格下位概念；本轮按 relation-only / parentless 概念处理，后续若要写 `up` 需要走概念层级候选生成、判定和 dry-run。

## 现代性状态

TTL 是基础工程概念 + 当前工程实践。

- 基础地基：有限生命周期、过期、刷新和清理是缓存、网络和分布式系统的长期稳定概念。
- 当前工程实践：RAG、Agent memory、LLM gateway、prompt cache、向量索引和知识库治理都会吸收 TTL 思想，用来处理 freshness、成本和风险。
- 易变部分：具体系统如何做 stale-while-revalidate、软删除、降权、后台刷新、用户确认和审计，是产品和框架实现细节。

## 现代系统怎么吸收 TTL 的价值

现代系统通常不会只用一个固定 TTL 解决所有问题，而是按信息类型分层：实时数据用短 TTL，稳定文档用长 TTL，高风险政策知识加版本和人工审核，用户偏好记忆加最后确认时间和可撤销机制，检索缓存加命中率和更新时间。

更成熟的系统还会记录“为什么这条信息仍可信”：来源、更新时间、版本、权限、最后命中时间、最近验证结果和责任人。这样 TTL 才不是一个孤立数字，而是 freshness governance 的一部分。
